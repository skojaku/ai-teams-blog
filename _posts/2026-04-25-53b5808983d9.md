*TL;DR: MCP and A2A are not rivals — they're complements, and this paper pins down the complexity crossover where multi-agent delegation overtakes single-agent tool integration, thanks to a 3.1× reduction in context-window bloat.*

## When One Agent Chokes on Its Own Context

A single LLM agent fields a question about three open-source projects. It calls the GitHub API and stuffs the response into its context window. Calls npm—stuffs that. Calls OSV.dev for vulnerabilities—stuffs that too.

By the time it reaches Stack Overflow, the window is carrying 34,959 tokens. Prompt, completion, and every accumulated tool response, most of it irrelevant to the current step.

That's not a hypothetical. That's exactly what happened in this benchmark.

Ivan Dobrovolskyi ran 450 controlled experiments across three architectures and found a clean crossover: single-agent MCP is faster for simple queries, but A2A wins on complex ones. The culprit is context-window bloat, and this is the first paper to quantify it.

## Two Protocols, Two Jobs

The Model Context Protocol (MCP) lets one agent call external tools through stdio or HTTP, keeping everything in a single context window. It's the standard for *agent-to-tool* integration — Anthropic released it in November 2024, and the SDK now exceeds 97 million monthly downloads.

The Agent-to-Agent Protocol (A2A) lets autonomous agents discover and delegate to each other over HTTP via JSON-RPC. Each agent publishes an Agent Card at `/.well-known/agent.json`, lays claim to specific capabilities, and maintains its own context. Google launched A2A in April 2025 with 50+ partners on board.

Ehtesham et al. (2025) surveyed MCP, ACP, A2A, and ANP and argued they're complementary layers — but their analysis was qualitative. No measurements. No controlled experiments. This paper fills that gap.

## The Setup: Three Architectures, One App

Dobrovolskyi built an open-source "Health Analyzer" that answers natural-language queries about open-source projects by pulling live data from four public APIs: GitHub (repository stats), npm (download counts), OSV.dev (vulnerability data), and Stack Overflow (community metrics).

The same application logic was implemented three ways:

- **MCP-only:** A single Claude Sonnet agent connects to four stdio tool servers. Every tool response piles into one context window.
- **A2A multi-agent:** A coordinator agent delegates to four specialist agents over HTTP. Each specialist carries roughly 2,263 tokens in its own lean context.
- **Hybrid:** A zero-cost keyword heuristic classifies query complexity at the door and routes to MCP for simple queries, A2A for complex ones.

Thirty queries span three tiers — ten simple (single-source, single-project), ten medium (multi-source, single-project), ten complex (multi-source, multi-project comparison). Each query ran five times per architecture. All 450 executions used the same model: `claude-sonnet-4-20250514`.

Six metrics per run: wall-clock latency, total LLM tokens, cost in USD, LLM API calls, data-source API calls, and code complexity (LOC + cyclomatic complexity). The stats are rigorous — bootstrap 95% CIs with 10,000 resamples, Mann–Whitney U with Bonferroni correction for nine simultaneous tests, and Cliff's delta effect sizes.

## The Crossover

Here's the headline result, broken down by complexity tier.

**Simple queries.** MCP finished in 8.9 seconds. A2A took 19.8 seconds. The difference is large and unambiguous ($p < 0.0001$, $\delta = -0.90$). MCP's advantage is zero overhead: no HTTP connections, no agent discovery handshake, no delegation LLM call. A2A pays that overhead on every query, even trivial ones.

**Medium queries.** MCP still leads at 30.3 s vs A2A's 36.0 s ($p = 0.0001$, $\delta = -0.51$). But the gap is narrowing. A token gap is opening — MCP uses 9,965 tokens, A2A uses 6,420 — and it will invert at the next tier. There's a wrinkle, though: A2A costs *more* on medium queries ($0.0521 vs $0.0435), because the multi-agent protocol's extra LLM calls generate a larger share of output tokens. Fewer tokens, but more expensive ones.

**Complex queries.** A2A flips the script: 45.1 s vs MCP's 51.8 s ($\delta = +0.29$, small-to-medium effect). And the token numbers tell the real story. MCP gorges on 34,959 tokens — 3.1× what A2A consumes at 11,318 tokens ($p < 0.0001$, $\delta = +0.84$). Every tool response the single agent has ever received is still sitting in its context window, growing the prompt on each successive API call. A2A's specialists each carry a fraction of that burden.

Cost follows tokens directly. A complex query costs $0.130 on MCP vs $0.079 on A2A — a 39% reduction. Scale that to a thousand complex queries and you're comparing $130 to $79.

## Why the Crossover Happens

Two forces oppose each other.

MCP's overhead is *constant and low* — a single process, no networking, no delegation step. That makes it fast out of the gate.

A2A's overhead is *fixed and front-loaded* — you pay for HTTP connections, Agent Card discovery, and a coordinator LLM call regardless of query difficulty. But A2A's overhead does *not* scale with complexity, because each specialist agent starts fresh.

MCP's cost *scales with the number of tool calls*, because every response accumulates in one growing window. The more APIs a query touches, the worse the bloat.

So the crossover is structural: `simple-query MCP advantage (constant low overhead)` → `multi-tool-query A2A advantage (context isolation wins)`.

![Line chart showing the complexity-dependent crossover where MCP outperforms on simple and medium queries while A2A overtakes on complex queries]({{ "/assets/figures/53b5808983d9/fig_003_02.png" | relative_url }})

The hybrid architecture confirms this. A cheap keyword router — no LLM call, just pattern matching — routes simple queries to MCP and complex ones to A2A. It tracks the winner at every tier with near-optimal latency, tokens, and cost.

## The Tradeoff: Code Complexity

MCP wins on implementation simplicity. It requires 706 LOC across 11 files, with average cyclomatic complexity of 7.9. A2A demands 1,530 LOC across 15 files (complexity 10.7) — 2.2× more code, largely due to HTTP server infrastructure (291 LOC) and per-agent boilerplate. The hybrid adds a routing layer on top, reaching 1,899 LOC.

If you're building a quick tool-call pipeline, MCP is the path of least resistance. If you're orchestrating complex multi-source workloads, A2A saves you tokens and money — but you pay upfront in engineering effort.

## What This Doesn't Tell Us

All experiments ran on a single LLM model — Claude Sonnet. Different models, different token pricing, different context window sizes could shift the crossover point. The A2A agents ran on localhost, eliminating network latency entirely; real distributed deployments would add round-trip overhead that favors MCP. And the benchmark covers one domain (open-source project analysis) with one set of four APIs. Other domains might move the numbers.

Ehtesham et al. hypothesized that MCP and A2A are complementary. Previous frameworks like AutoGen (Wu et al., 2023), CrewAI, and MetaGPT all built proprietary communication layers — none used standardized protocols. Toolformer (Schick et al., 2023) showed that LLMs can learn to invoke tools during generation, which is the capability MCP standardizes. But until now, no one had measured head-to-head what happens when you actually deploy these protocols at different levels of complexity.

The 450-execution benchmark is open-source under MIT. All runs completed with zero execution failures, and numerical claims in agent outputs were verified programmatically against live API responses with no divergences detected.

## When One Window Isn't Enough

*Simple query → MCP. Complex query → A2A. Unknown complexity → hybrid router.* The crossover is real, measurable, and structural — it comes down to whether your agent's context window is an asset or a liability. When the task outgrows one window, it's time to split.
