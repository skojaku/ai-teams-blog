*TL;DR: The Model Context Protocol quietly burns 50k+ tokens per turn re-serializing every tool schema, but a middleware layer called Tool Attention cuts that by 95% by loading only the tools the agent actually needs—using intent-conditioned gating instead of eager injection.*

## Before the User Even Speaks

Six MCP servers, 120 tools, 30 turns. Over that session, the protocol re-serializes every tool's JSON schema on every single turn—burning roughly 1.42 million tokens that add nothing to the agent's reasoning.

That is the Tools Tax. Not a rounding error—an order-of-magnitude cost amplifier, a reasoning poison, and an attack surface, all in one.

Sadani and Kumar identify this tax as a protocol-design artifact, not an inevitability. The cure: treat the tool catalog the way transformers treat tokens—let each turn dynamically attend only to what matters.

They call it Tool Attention.

## Why the Tax Hurts More Than You Think

The MCP specification is stateless. Every turn, the host re-serializes the entire tool catalog into the prompt.

No memo, no diff, no "you already know this." Fresh copies of every schema, every time.

Three cascading failures follow.

**Economic.** One published benchmark shows CLI-equivalent workflows costing $3.20 versus MCP at $55.20 for the same 10,000 operations. FinOps audits consistently find schema tokens responsible for 40–60% of total agent API spend.

**Cognitive.** Once context utilization crosses roughly 70%, LLM reasoning quality falls off a cliff. Models hallucinate parameters, confuse similar tools, lose the thread. With 120 tools, you hit that fracture point almost immediately.

**Adversarial.** Every injected description is parsed by the model. A single poisoned schema can hijack the agent—whether or not the tool is ever invoked. More schemas, more attack surface.

## From Tokens to Tools: The Attention Analogy

Vaswani et al. replaced recurrence with self-attention: instead of squeezing every token through a fixed-width hidden state, let every position attend only to what it needs.

Tool Attention applies the same logical move one level up. Instead of dragging the full catalog through the prompt regardless of relevance, let each turn select a small, intent-conditioned subset.

The theoretical grounding comes from MindGuard (Wang et al., 2025), which defined Total Attention Energy (TAE)—a metric showing that successful tool calls accumulate high attention energy between the generated action tokens and the relevant schema tokens.

Tool Attention exploits the contrapositive: tools whose expected TAE is negligible can be excluded without changing the outcome.

Toolformer (Schick et al., 2023) represents the old pattern being replaced—treating the entire tool set as fixed and fully injected, exactly the design that produces the tax at scale.

## How Tool Attention Works

The mechanism has three components stacked inside a middleware `before_model` hook.

### Intent–Schema Overlap (ISO)

Each tool gets a compact summary—60 tokens or fewer—pre-embedded with `all-MiniLM-L6-v2` (384 dimensions). At each turn, the user query gets the same embedding treatment.

The Intent–Schema Overlap is the cosine similarity:

$$\text{ISO}(q, t_i) = \frac{e_q^\top e_{t_i}}{\|e_q\|_2 \, \|e_{t_i}\|_2}$$

Tools below a calibrated threshold θ (typically 0.22–0.32) are filtered out. No semantic match, no schema injection.

### State-Aware Gating

On top of the ISO score, the gating function enforces deterministic preconditions: does the agent hold the right auth token? Has a prior tool's output already landed?

These are hard Boolean checks on authoritative state—not free-text heuristics that an adversarial paraphrase can bypass.

$$g(t_i; q, \text{state}_t) = \mathbb{1}[\text{ISO}(q, t_i) \geq \theta] \cdot \mathbb{1}[\text{state}_t \models \text{pre}_i]$$

### Two-Phase Lazy Loading

Even after gating, naively injecting full JSON schemas for ten tools costs 2–4k tokens. The lazy loader splits injection in two:

- **Phase 1 (always resident):** All 120 compact summaries stay in context—about 4.8k tokens, stable across turns, cache-friendly. The model always knows what tools exist.
- **Phase 2 (per-turn, on demand):** Only the top-*k* gated tools get their full schemas promoted from an out-of-context registry.

The layout yields an 84% prompt-cache hit rate versus 22% for full-schema injection.

### The Hallucination Gate

Aggressive gating creates a risk: the model might try to call a tool it saw in the summary but whose schema was never promoted. A deterministic `after_model` hook catches this.

If the model calls a tool not in the active set, the call is rejected with a structured error listing available alternatives.

The gate triggers on only 2.3% of turns. In 78% of those cases, the model recovers on the next try. Zero unrecoverable failures observed.

## What the Numbers Show

The benchmark: 120 tools, six servers, per-server token counts calibrated to public deployment audits. Token counts measured directly with `tiktoken`. Downstream success, latency, and cost figures are projections—the authors flag this explicitly throughout.

The headline:

- **Per-turn tool tokens: 47,312 → 2,368.** A 95.0% reduction, measured directly.
- **Effective context utilization at turn 30: 0.24 → 0.91.** The agent keeps 91% of its context for reasoning—far from the ~70% waste threshold where performance collapses.
- **Projected task success: ~72% → ~94%**, a ~22 percentage-point gain.
- **Projected P50 latency: ~52% reduction**, ~86% cost reduction.

Ablations reveal the lazy loader as the single biggest contributor (+10.3 pp success), with stateful preconditions adding +3.6 pp and the hallucination gate adding +3.2 pp. Swapping MiniLM for TF-IDF costs 8.1 pp—semantic matching matters.

A defensive side effect: gating excludes 46 out of 50 simulated poisoned tool descriptions, reducing projected Tool Poisoning Attack success from 38% to 6%. Not a targeted defense, but a significant free reduction in attack surface.

## Where It Breaks

The honest limits.

All downstream metrics—success, latency, cost, reasoning quality—are projections from measured token counts combined with published deployment telemetry. The authors did not run 500 live tasks against a paid LLM API. The token reduction is the primary empirical claim; everything else is well-motivated extrapolation.

Retrieval quality hinges on tool summary quality. Cryptic names and vague descriptions hurt F1. The mechanism cannot fix protocol-level deficiencies like missing capability negotiation.

Multi-hop workflows where the correct tool only becomes relevant after an intermediate result cause about 17% of failures. Re-embedding the query after each observation partially mitigates this but is not yet integrated.

An adversarial tool description engineered to cosine-match a benign query will slide through the gate. The authors recommend pairing Tool Attention with MindGuard's TAE monitor for runtime detection.

## The Real Lesson

The Tools Tax is not a cost-of-goods problem. It is a protocol-design choice—stateless, eager, uniform injection—that becomes pathological at scale. Tool Attention shows that a model-agnostic middleware layer, deployable today on stock MCP infrastructure, can cut the tax by 95% without changing the protocol.

The deeper point: context engineering, not raw context length, is the binding constraint on agentic systems. Every schema you inject is a tax on attention. Gate ruthlessly.

Tool attention, in other words, is all you need.
