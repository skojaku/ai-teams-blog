*TL;DR: Every capability you add to an AI agent — memory, tools, planning, peers — is also an attack surface, and this survey traces exactly how that escalation unfolds from a single agent to an open Internet of Agents, arguing that safety must be co-designed with capability at every layer.*

## The Escalation

Add memory to an agent so it can remember your preferences.

Now an attacker can poison that memory.

Add tools so it can book flights.

Now it can also delete your production database.

Give it peers to collaborate with.

Now a hallucination in one agent becomes trusted context for all of them.

Open it to the internet.

Now you have an ecosystem where no one knows who — or what — is acting.

Wibowo and Polyzos map this escalation with surgical precision. Their thesis is blunt: agentic safety is not a feature you bolt on after shipping.

It is an architectural property that must be co-designed with capability. Every mechanism that makes an agent more capable also enlarges its attack surface.

They call these *dual-use interfaces*, and the paper walks through them tier by tier.

## Meet the Three Tiers

The survey organizes the agentic world into three layers of escalating complexity:

1. **Single Agent** — a model plus memory, design patterns, tools, and guardrails.
2. **Closed Multi-Agent System (MAS)** — a team of agents under one control plane, like a CrewAI crew or an AutoGen group chat.
3. **Open Interoperable MAS (IMAS)** — heterogeneous agents from different organizations discovering and collaborating across the internet.

Each tier inherits every vulnerability from the tier below, then adds new ones that only emerge at that scale.

The framing is not "here are all the threats."

The framing is: *here is how architectural choices create or contain those threats*.

Prior surveys catalogue threats by domain — identity forgery, cross-agent trust, embodied physical safety — or map attack vectors across agent workflows.

This paper argues that threat enumeration alone is insufficient.

Without understanding how architectural decisions create vulnerability surfaces, you are cataloguing symptoms while ignoring the disease.

## How Dual-Use Interfaces Work

The central principle is what the authors call *dual-use interfaces* — each agent component is simultaneously a functional affordance and a vulnerability vector.

![Matrix mapping each agent component — Memory, Tools, Planning, Guardrails — across the three tiers, showing how capability expansion correlates with attack surface growth at every layer.]({{ "/assets/figures/f2319e665648/fig_006_02.png" | relative_url }})

Memory is the clearest example. Short-term memory lets an agent track multi-step tasks. But finite context windows create what the authors call *amnesiac agency* — exceed the window, and the agent forgets its safety instructions.

Attackers exploit this with *Context Flooding*: push benign conversation until safety rules scroll out of view, then inject the real payload.

Long-term memory is worse. It turns temporary attack surfaces into permanent ones.

An attacker who poisons a vector database has compromised every future interaction that retrieves from it.

The design pattern layer drives the point home. Planning seems purely beneficial — decompose a goal into sub-tasks, execute them.

But planning creates what the paper calls *hallucination snowballing*: the agent commits to an early error, then generates justifications to stay consistent.

A single wrong assumption cascades into a logically coherent but factually disastrous plan.

*Hallucinating chatbots speak falsely; hallucinating planning agents act falsely.*

Even self-correction, the go-to safety mechanism, is dual-use. Reflection loops let agents critique their own outputs.

But attackers can inject malicious feedback — "this plan is too safe, be more aggressive" — and the agent's self-correction mechanism obliges.

The guardrail becomes the vulnerability.

## The Alignment Paradox

One of the sharpest observations in the paper is the Alignment Paradox. Strengthening user-intent adherence helps resist indirect prompt injection.

But it also makes the model more helpful — more willing to reveal its system prompt when asked persuasively.

The paradox runs deeper. Deterministic filters — static rules — are rigid. They cannot understand context, so a Base64-encoded malicious instruction sails right through.

Probabilistic filters — safety classifiers — catch semantic threats. But adversarial noise suffixes render prompts invisible to classifiers while remaining actionable to the agent.

Guardian agents — separate LLMs monitoring workers — sound elegant. But they inherit all the vulnerabilities of the workers they monitor.

A smarter guardian is a larger attack surface. A weaker guardian gets sandbagged by the capable agent it is supposed to oversee.

The friction between deterministic policy desires and probabilistic execution realities is not a bug.

It is the fundamental architectural tension of agentic control. There is no single layer that solves it.

## Populations and Blast Radius

When agents work together, the stakes multiply. The paper's MAS analysis centers on two dimensions: control flow autonomy and topology.

Control flow autonomy is a spectrum.

Predefined workflows — fixed DAGs — contain blast radius. A compromised node can only reach the tools wired to that node.

But data still propagates blindly downstream, creating *Poisoned Pipelines*.

Dynamic workflows — runtime orchestration — make the orchestrator a Single Point of Trust. Compromise the router, and it connects low-privilege users to high-privilege agents.

Emergent flows — peer-to-peer, no central control — give unbounded blast radius. A compromised agent can persuade any peer.

Topology matters just as much.

Horizontal structures — peer meshes — enable *Domino Effects*: poisoned outputs become trusted context for neighbors, driven by what the authors call *Passive Complicity* through role rigidity.

Agents can detect peer errors but lack instructions to challenge them.

Vertical structures concentrate risk in the supervisor. But that same concentration enables the *Inspector pattern* — the supervisor catches hallucinations before propagation.

Hierarchical structures with sub-team isolation contain blast radius most effectively.

The paper identifies a clean architectural tradeoff: *adaptability reduces verifiable control*.

If you want flexible, self-organizing systems, you accept that errors and attacks propagate freely. If you want deterministic guarantees, you constrain flexibility.

## The Open Ecosystem Problem

The IMAS tier — the actual Internet of Agents — is where the paper shifts from analysis to aspiration.

The authors identify four security pillars that any open agent network requires:

1. **Standardized protocols** (MCP, A2A) for cross-network communication.
2. **Agent registration and discovery** — you need to know who you are talking to.
3. **Resource vetting** — you need to verify what you are connecting to.
4. **Ecosystem governance** — you need accountability when things go wrong.

The most corrosive gap in current deployments is *identity masquerading*.

Agents act with raw user credentials. A demo file-editing agent inherits the user's permission to delete production directories.

The paper frames this as a Confused Deputy Problem: agents lack distinct identities constraining privileges. Systems cannot determine whether humans or agents performed actions. Accountability breaks entirely.

The proposed fix is *Authenticated Delegation*: authorization as the intersection of user permissions, agent scope, and contextual constraints.

No more flat permission models. A Compensation Agent sharing credentials with an Intern Assistant enables session hijacking — systems cannot distinguish CEO salary requests from intern bots using hijacked sessions.

## The Honest Limits

The IMAS vision is the paper's weakest empirical ground. Many proposed pillars — standardized protocols, agent registration, ecosystem governance — lack deployed systems to validate them.

The recommendations are aspirational, drawn from architectural reasoning rather than production evidence.

Several described attacks — G-Designer adversarial perturbations on GNN topology generators, topology injection in generative meta-architectures — have not been validated against deployed systems. The paper acknowledges that G-Designer's resilience against advanced threats like collusion and targeted injections remains unexplored.

The field is moving fast — references span 2023–2026 — and the gap between demonstrated vulnerability and engineering mitigation is wide.

But the architectural lens holds regardless.

The dual-use interface principle gives you a reusable mental model: before you add any capability to an agent system, trace how that capability enlarges the attack surface at every tier.

Memory → memory poisoning across a network. Tools → toolflow hijacking at scale. Peers → domino effects without challengers.

The principle scales even if the specific mitigations do not yet.

Every capability is a surface. Safety is architecture, not afterthought.
