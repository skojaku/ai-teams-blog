*TL;DR: Memory, skills, and rules aren't three separate research problems—they're three compression levels on the same axis, and the two communities working on them cite each other less than 1% of the time.*

Two communities study how LLM agents learn from experience. One builds memory systems. The other builds skill systems. Across 1,136 references in 22 primary papers, they cite each other less than 1% of the time.

Memory papers cite skill work at 0.7% (4 out of 566). Skill papers cite memory work at 1.2% (7 out of 570). Neither skill survey mentions any memory system. Only one memory survey mentions any skill work—and that single citation is Voyager.

Both communities compress the same raw material—interaction traces—into reusable knowledge. Both independently solve retrieval, conflict detection, staleness, and evaluation. Without talking to each other.

This paper reframes the disconnect: memory, skills, and rules aren't three problems. They're three compression levels on a single axis.

## How the Spectrum Works

Four levels of experience compression, each more abstract than the last:

- **L0 — Raw trace.** The full interaction log. Compression ratio: 1:1.
- **L1 — Episodic memory.** Structured event summaries that preserve key context while discarding redundant mechanics. Compression ratio: 5–20×. Ten surveyed systems cluster here.
- **L2 — Procedural skill.** Reusable behavioral patterns—code snippets, workflow templates—that abstract across instances. Compression ratio: 50–500×. Eight systems cluster here; Voyager pioneered the paradigm.
- **L3 — Declarative rule.** Domain-invariant principles and constraints. Compression ratio: 1,000×+. Nearly empty—no surveyed system automates extraction at this level.

![The Experience Compression Spectrum: memory, skills, and rules as progressively compressed outputs from the same interaction traces]({{ "/assets/figures/eaeafa6ad3da/fig_002_01.png" | relative_url }})

The ratios tell an efficiency story. An L1-only agent storing 1,000 episodes at ~500 tokens each maintains a ~500K-token knowledge store it must search at every decision.

Compress to L2 skills: ~5K tokens. Compress to L3 rules: ~500 tokens.

That compounding reduction matters when an agent makes thousands of decisions per day.

Critically, these aren't sequential pipeline stages. A system could compress directly from L0 to L3, or maintain knowledge at multiple levels simultaneously. The spectrum describes the output space, not a fixed processing order.

## The Missing Diagonal

Map 20+ systems onto this spectrum and a pattern jumps out.

Ten systems cluster at L1. Eight cluster at L2. Only two—ExpeL and AutoAgent—operate at two levels simultaneously.

Both use predetermined levels. Two-speed systems, not continuous spectra.

Nothing supports adaptive level selection. Nothing promotes knowledge upward when patterns accumulate, or demotes it downward when abstraction fails.

The authors call this the *missing diagonal*—the absence of systems that dynamically choose the right compression level for a given trace, promote recurring memories into skills, or demote a rule that's too abstract for the current context.

This isn't an engineering gap. It's a framing problem.

Each community defines its output format *a priori*. Memory papers build memory. Skill papers build skills.

Nobody asks: given this trace, what *kind* of knowledge should I extract?

## Higher Compression Beats Lower

The empirical evidence is consistent, if pieced together from separate studies:

- SkillRL: L2 skills outperform L1 trajectory retrieval by **+68.5pp** on ALFWorld.
- Trace2Skill: outperforms human-written skills by +21.5pp on SpreadsheetBench; beats no-skill baselines by +42.1pp.
- EvoSkill: L2 skills transfer across tasks (+5.3% on BrowseComp).
- Trace2Skill: skills transfer across model sizes (35B → 122B, +57.7pp).

Transferability increases with compression. L1 memories transfer across base models. L2 skills transfer across tasks and model sizes. L3 rules, by the same logic, would transfer across domains—though no system exists to test this yet.

But compression level alone isn't enough. SkillsBench finds that curated skills help (+16.2pp) while LLM-self-generated skills provide exactly zero benefit (+0.0pp).

Compact noise is still noise.

## The L3 Desert and a Surprising Signal

Level 3 is almost entirely empty. No surveyed system automates rule extraction from agent experience.

Constitutional AI uses pre-specified rules. Reward-shaping methods encode L3-type knowledge, but it's human-designed, not learned. Weight-level rules via RLHF are frozen after training, opaque, and can't be updated without retraining.

The barriers are real.

Distinguishing causal regularities from spurious correlations is hard.

Rules without L2 grounding risk being too abstract.

No methodology exists for evaluating rule quality.

But one recent study lights a path. RuleShaping analyzes 25,000+ natural-language rules for coding agents and finds that **negative constraints improve performance by 7–14pp while positive directives hurt**.

The implication: if you're going to compress to L3, compress into guardrails—*don't do this*—not directives—*do this*.

## What the Spectrum Reveals

View the landscape through the compression lens and four structural insights appear—none visible from within either community alone.

**Specialization alone is insufficient.** Both communities independently solve retrieval, conflict detection, staleness, and evaluation without sharing solutions. A unified architecture with shared infrastructure would avoid redundant engineering.

**Evaluation is level-coupled.** L1 systems optimize for QA F1. L2 systems optimize for task success rate. L3 has no established metric.

A better framework would measure downstream impact per token of stored knowledge, regardless of compression level.

**The transferability–specificity trade-off follows a concave curve.** L2 likely sits at the sweet spot, balancing broad applicability with actionable context.

**Lifecycle management is an afterthought.** Merging skills doesn't always help—success-trace signals can cause ±21pp performance swings.

No system addresses the full lifecycle: conflict detection, staleness recognition, principled deprecation, and cross-level consistency.

## Building the Diagonal

The authors sketch a concrete architecture. A meta-controller routes each trace to one or more level-specific compressors based on novelty and frequency.

A first-seen pattern stays at L1. Once *k* similar entries accumulate, a promotion engine consolidates them into an L2 skill. After cross-domain recurrence, the engine generalizes to an L3 rule.

Demotion works in reverse. When a skill is repeatedly retrieved but unused, or when a rule fails in a novel context, the system drops back to L1 for fresh evidence collection.

A lifecycle manager tracks provenance, confidence, usage frequency, and last validation time for every artifact. Upward compression—consolidation—happens during idle periods, like hippocampal-neocortical consolidation during sleep.

No current system does this. All compress synchronously at ingestion.

Picture a customer-support agent. It hits a timeout on `/api/export` and stores an L1 memory.

After five similar episodes, promotion produces an L2 skill: *HANDLE EXPORT TIMEOUT—check batch size, reduce if >1000 rows, retry*.

After dozens of instances across endpoints, generalization yields an L3 rule: *Timeouts on data-intensive endpoints typically stem from oversized batches*.

The rule fails in a novel context? Demotion drops back to L1.

## The Honest Limits

The framework is conceptual, not empirically validated. The spectrum's utility as a design tool awaits experimental confirmation.

The four discrete levels are a simplifying abstraction. Compression may be continuous in practice—EvolveR's strategies straddle L2/L3.

The survey is a snapshot of Q1 2026. The field moves fast.

The testable predictions—multi-level stores should outperform single-level as deployment length increases, L3 constraints should beat directives—haven't been tested yet.

Validating or falsifying them would establish whether this framework is a useful design tool or merely a tidy taxonomy.

Two communities, less than 1% cross-citation, one compression spectrum. The frame that makes the gap visible is the first map for closing it.
