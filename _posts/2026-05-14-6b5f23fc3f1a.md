*TL;DR: You can replace 60,000 lines of deterministic backend code with an LLM that infers state on the fly — and still get reliable, state-based evaluation of multi-turn tool-calling agents, with 94.7% human–LLM agreement on hallucination detection and near-zero simulator hallucination.*

AppWorld needed roughly 60,000 lines of code for its deterministic engine, another 40,000 for the benchmark, plus 1,800 unit tests written over 14 months.

That is the price of evaluating a multi-turn, tool-calling agent the rigorous way: build a real database backend, run the agent against it, check the final state. This paper asks a pointed question — what if you skip the database entirely?

## The Tax on Deterministic Backends

State-based evaluation — checking whether the final database state and the agent's last response match expectations — is the right idea. Multiple tool-calling paths can correctly solve the same user goal, so trajectory matching is too brittle.

τ-bench and τ²-bench proved this principle with deterministic backends.

AppWorld went further, building a full controllable world of apps.

But every one of these benchmarks pays the same tax: you need a working, correct, deterministic environment before you can evaluate a single agent. That means schema design, deterministic tool implementations, and assertions — months of engineering before the first rollout.

Proxy State-Based Evaluation asks: can we keep the rigor of final-state checking while ditching the deterministic backend?

## Scenarios Replace Trajectories

**Move one:** replace ground-truth trajectories with *scenarios*. A scenario $z$ encodes a user goal $g(z)$, user facts $u(z)$, system facts $s_0(z)$, an expected final state $s^*(z)$, and expected agent behavior $b^*(z)$. The scenario says *what* should happen, not *how*. Multiple correct paths are treated equally.

**Move two:** replace the deterministic database with an LLM *proxy state tracker*. At each step $(t, k)$ of the interaction, the state tracker infers the current proxy state $\tilde{s}_{t,k} = f_\text{state}(\tau_{\leq(t,k)})$ from the full conversation trace — not from reading a real database. Tool simulators "read from" and "write to" this proxy state. After the conversation ends, an LLM judge compares the proxy final state against the scenario's expected final state.

This is the key shift: *deterministic backend* → *LLM-inferred proxy state*. The evaluation still checks final-state conditions, but the state is reconstructed from the trace rather than maintained by 60K lines of Python.

## How Five LLM Roles Cooperate

The benchmark-as-environment has five LLM roles:

1. **Reasoning Agent (RA):** the model under test. It follows a ReAct loop — reason, call a tool or show a message, observe the response, repeat.

2. **User Simulator ($f_\text{user}$):** generates the next user utterance conditioned on the scenario's goal, facts, and a persona (power, ambiguous, or confused).

3. **Tool Simulators ($f_\text{tool}$):** return structured JSON outputs for `call_account(q)` and `call_commerce(q)`, conditioned on the initial system facts and the current proxy state.

4. **State Tracker ($f_\text{state}$):** infers proxy state transitions after each tool call, categorizing operations as reads (no state change) or writes (state change only on success).

5. **LLM Judges ($J_\text{goal}$, $J_\text{hall}$):** one judge scores goal completion; the other flags tool-side and user-side hallucinations against the scenario constraints.

The state tracker conditions on the *full* trajectory prefix rather than incrementally updating from the previous state. This is a deliberate design choice — it avoids error accumulation in long conversations.

## What the Numbers Show

The benchmark includes 208 scenarios across commerce and account management (157 train, 51 test), each run for 10 independent rollouts.

**Goal completion scales with model strength.** GPT-5 at high reasoning effort hits 85.76% GC. GPT-4o reaches 66.04%. GPT-4o-mini drops to 59.83%.

Within the GPT-5 family, GC increases monotonically with reasoning effort: high (85.76%) > medium (80.53%) > low > minimal. Rankings hold even when you swap the entire simulator-and-judge stack from GPT-5o to Gemini-2.5-Pro.

![Goal completion rate across baseline reasoning agents and trained models, showing monotonic scaling with model strength and reasoning effort]({{ "/assets/figures/6b5f23fc3f1a/fig_006_02.png" | relative_url }})

**SFT on off-policy data transfers.** Training Qwen3-30B-A3B on successful rollouts from a GPT-5 teacher boosts GC from 65.64% to 77.34% — an 11.7 percentage-point gain. RFT yields a more modest 67.11%. Using a Qwen3-235B teacher instead still improves the base model from 65.64% to 70.87%, confirming the gain is not an artifact of GPT-5-specific data.

**Long conversations are where agents break.** Base-model GC drops from 74.75% on 1–2 turn conversations to 42.75% on 5+ turn conversations.

![Goal completion rate stratified by conversation turn depth, showing base model performance collapse on long conversations and SFT recovery]({{ "/assets/figures/6b5f23fc3f1a/fig_014_04.png" | relative_url }})

SFT recovers the long-conversation bucket by +26.69 percentage points — the biggest absolute gain. Multi-turn depth is the real stress test.

**Hallucination stays near zero under the default configuration.** With GPT-5o at medium reasoning as the simulator and state tracker, tool hallucination is 1.33% and user hallucination is 0.67%. Three-way agreement (two humans + LLM judge) reaches 82.7% on goal completion and 94.7% on both hallucination dimensions.

## Honest Limits

The proxy state is *inferred*, not deterministic. When you swap the state tracker to a weaker model (GPT-4o), tool hallucination climbs from 1.33% to 3.61%.

Residual evaluation error exists — the question is whether it is small enough to tolerate. The paper makes a strong case that it is, but the answer depends on your domain.

The scenarios are synthetic and limited to e-commerce and account management. Whether this framework generalizes to safety-critical or open-ended domains — medical, legal, financial advisory — is unvalidated. A state tracker that hallucinates 1.33% of the time in a shopping context might be unacceptable when the stakes are higher.

Then there is the circularity question: you are using LLMs to simulate the environment *and* to judge the results. The cross-model robustness experiments help — rankings survive when simulators and judges come from a different model family — but the validation remains empirical, not formal.

## Why This Matters

This work lowers the engineering barrier for building agentic benchmarks by an order of magnitude.

Instead of months of building deterministic backends, you write scenario specifications and let LLMs handle the simulation. The 94.7% human–LLM agreement on hallucination detection and near-zero simulator hallucination suggest you lose very little reliability in the trade.

More practically, the framework produces *training data*, not just test scores. Off-policy rollouts from a strong teacher, filtered by the goal-completion judge, give you SFT data that transfers to unseen scenarios. That is the loop that matters for industry: benchmark → train → evaluate → iterate.
