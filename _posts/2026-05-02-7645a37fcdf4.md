*TL;DR: A separate reviewer agent that inspects tool calls before they execute—not after—boosts multi-turn agent accuracy by over 7 percentage points without retraining the base model, and a new Helpfulness-Harmfulness metric shows reasoning-model reviewers give a 3:1 benefit-to-risk ratio.*

## Before the Call Goes Out

An agent calls `get_weather("NYC", unit="C")`.

The call goes through. The user gets Celsius when they expected Fahrenheit, and "NYC" violates the API's naming convention. The error has already happened.

Undoing it means maintaining state, replaying context, hoping the model can course-correct. That's the state recovery problem, and it haunts every tool-calling agent.

Now imagine a second agent looking over the first one's shoulder before the call goes out. It spots the abbreviation and the unit issue, injects feedback as a system message.

The first agent revises to `get_weather("New York City", unit="F")`. No damage. No recovery needed.

Anh Ta, Junjie Zhu, and Shahin Shayandeh at Apple built that second agent—and showed it works.

## Why It Matters

Tool-calling agents routinely select the wrong tool, mis-specify arguments, or fail to recognize when no tool applies at all.

Self-correction approaches like Self-Refine and Reflexion make the same model generate *and* critique its own calls. When that model makes an irreversible mistake—deleting an alarm instead of updating it—self-correction becomes state recovery. Context windows can't always hold the undo.

The core shift: separate execution from review at inference time.

Error correction turns from a post-hoc recovery problem into a proactive prevention problem.

## How the Duo Works

A primary tool-calling agent (GPT-4o in all experiments) generates a provisional tool call. A separate reviewer agent evaluates it before execution.

If the reviewer finds errors, it injects feedback as a system message and the primary agent revises.

The loop continues until the reviewer approves or a cap is hit.

![Progressive Feedback architecture showing the reviewer evaluating provisional tool calls before execution, with feedback loops for revision]({{ "/assets/figures/7645a37fcdf4/fig_003_01.png" | relative_url }})

They test three collaboration mechanisms:

**Progressive Feedback (rN):** Iterate up to N revision cycles. The reviewer gives targeted feedback each round.

**Best-of-N Selection (sN):** Generate N candidates at varying temperatures, then let the reviewer pick the best one.

**Best-of-N Grading (gN):** Same as selection, but the reviewer assigns explicit numeric scores (0.0–1.0) with rationales. The highest-scored candidate wins.

Progressive Feedback dominates. It outperforms both Best-of-N variants by +4–5% on irrelevance detection, because it explicitly identifies and corrects errors rather than just ranking alternatives.

## Measuring the Tradeoff

Adding a reviewer isn't free. It can correct real errors—*helpfulness*—but it can also break correct calls—*harmfulness*. No prior work has quantified this tradeoff systematically.

$$\text{Helpfulness} = \frac{\text{cases where reviewer corrects a base agent error}}{\text{total base agent errors}}$$

$$\text{Harmfulness} = \frac{\text{cases where reviewer breaks a correct call}}{\text{total correct calls}}$$

$$\text{Benefit-to-Risk Ratio} = \frac{\text{Helpfulness}}{\text{Harmfulness}}$$

With GPT-4o as reviewer (v1 prompts), the ratio sits at 2.1:1. Swap in o3-mini—a reasoning model—and it jumps to 3.1:1 (36.8% helpfulness, 11.7% harmfulness).

The reasoning model's systematic verification reduces over-correction. Better reviewer, fewer false fixes.

## The Numbers

On BFCL (single-turn function calling), the best configuration reaches 92.5% relevance accuracy and 90.4% irrelevance detection—up from 90.9% and 84.9% at baseline.

That's a +5.5% gain on irrelevance. Exactly the category where agents struggle most.

![BFCL benchmark results across configurations, showing Progressive Feedback with GEPA-optimized prompts achieving the highest relevance and irrelevance scores]({{ "/assets/figures/7645a37fcdf4/fig_005_02.png" | relative_url }})

On τ²-Bench (multi-turn, stateful scenarios across airline, retail, and telecom domains), Progressive Feedback lifts average score from 48.7% to 55.8%—a +7.1% gain.

The biggest jump hits telecom: 41.2% → 64.0%.

But the reviewer doesn't just fix things. It introduces new error patterns.

Policy constraint violations drop by 13 percentage points, but "over-verbalization" errors jump by 17 points. The reviewer sometimes demands user-facing explanations where only a tool call is needed—an over-skepticism problem they trace through multiple rounds of prompt refinement.

## Evolving the Prompt

The reviewer's prompt turns out to be a first-class design surface.

They start with a minimal v1 prompt. They discover that 23% of review cycles are wasted on redundant "elaborate further" demands.

So they add a critical instruction: tool-only responses are complete—don't flag them as incomplete.

That cuts redundant loops from 23% to 8%.

Then they apply GEPA (Genetic-Pareto prompt evolution), which iteratively refines the reviewer prompt by reflecting on failure cases. The result: v3 prompts ~4.5× longer than hand-engineered v2 (1,599 vs. 358 tokens), packed with detailed error checklists and edge-case rules that manual engineering missed.

GEPA-optimized prompts add +1.5% on relevance and +2.8% on irrelevance on top of v2.

The lesson: a reviewer's prompt can be systematically improved—model by model, iteration by iteration—without ever touching the base agent.

## The Honest Limits

Only GPT-4o was tested as the base agent. Reviewer experiments spanned GPT-4o, o3-mini, and GPT-5 mini, but generalization to open-source models like Llama or Mistral remains unvalidated.

GEPA prompt optimization was applied only to BFCL (single-turn). On τ²-Bench, domain-specific v2 prompts sometimes *underperformed* the generic v1 prompts—manual tuning doesn't generalize across all tasks within a domain.

Latency stings on single-turn workloads: 6.2× overhead on BFCL (1.27s → 7.87s). On multi-turn episodes it's more palatable: 2.4× (158.7s → 384.3s), because reviewer cost amortizes across ~40 turns. They note that distillation into a smaller model could shrink this gap, but that's future work.

## Why You Should Care

Any team deploying a tool-calling agent can slot in a reviewer tomorrow—no retraining, no architecture changes, just one more inference call and a prompt.

The reviewer is independently optimizable: swap the model, tune the prompt, adjust the number of loops. The base agent stays untouched.

The Helpfulness-Harmfulness framework gives teams a principled answer to "is this reviewer actually helping?" before they ship it to production.

And the finding that matters most: catching errors *before* execution is fundamentally different from catching them after. The first prevents damage. The second attempts recovery.

Prevention wins.
