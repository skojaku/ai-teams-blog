*TL;DR: Instead of re-reading long skill instructions and growing histories at every step, this paper bakes recurring agent procedures into lightweight LoRA adapters and runs inference on constant-size context—matching or beating strong baselines while using 2–7× fewer prompt tokens per turn.*

By turn ten of a ReAct episode, the model has re-processed the same skill instructions ten times and is conditioning on thousands of tokens of accumulated history—just to pick the next action.

Xie et al. ask a blunt question: why not move that recurring procedure into the weights?

## Why Agents Re-Read Everything

Today's agent methods—ReAct prompting, memory-augmented retrieval, multi-turn RL training like DMPO and GiGPO—share the same inference habit. Before each action, the model conditions on task instructions, skill descriptions, retrieved memories, or accumulated histories.

For workflows you run daily, this is fundamentally wasteful.

The agent re-reads the same procedure at every turn, paying token costs that grow linearly with episode length. A personal assistant that re-learns "how to shop" on every click is burning compute on something it should already know.

The privacy angle sharpens the problem. Cloud models execute well but expose every intermediate state to external APIs. Local models keep data private but need every advantage to stay reliable. Both settings pay a token tax that grows with every step.

## From History to State

The key move is a change in what the model sees at inference time.

The old way: $c_t = \text{Format}(g, d_k, h_t)$, where $h_t$ contains all prior observations and actions—growing without bound.

The new way: $x_t = \text{Format}(g, o_t, q_t, b_t)$, where $o_t$ is the current observation, $q_t = (o_{t-1}, a_{t-1})$ is one step of context, and $b_t$ is a compact state block.

![Architecture contrasting history-based agent inference with constant-context agent inference using a tracker-rendered state block and a LoRA skill module]({{ "/assets/figures/4bd73e808358/fig_004_01.png" | relative_url }})

The state block is the decisive piece. It carries decision-relevant progress—acquired objects, completed subgoals, checked locations, remaining requirements—rendered by a deterministic tracker, not an LLM.

No growing history. No re-processed skill prompt.

The context size is bounded: $|x_t| \leq B_k$ for all $t$.

## How the Tracker Works

The tracker is not a neural network. It is a deterministic algorithm $M_k$ that parses actions and observations with lightweight rules and maintains a structured state $m_t$.

At each step: $m_t = \text{Update}_k(m_{t-1}, a_{t-1}, o_t)$, then $b_t = \text{Render}_k(m_t)$.

In ALFWorld, the block records the target object, whether the agent is holding it, the destination receptacle, and checked locations. In WebShop, it tracks the current query, inspected product, selected options, and purchase readiness. In SciWorld, it tracks the current phase, selected entity, inventory, and completed subgoals.

Zero model calls at inference. The tracker is cheap, fast, and reproducible.

## Baking the Procedure into Weights

Each task family gets its own LoRA adapter—about 0.5–0.7 GB on disk, updating roughly 2% of parameters. The base model stays frozen.

The pipeline has two stages.

### Stage 1: Imitate the Experts

Expert trajectories are replayed through the tracker to produce step-level input–output pairs: $(x_t, a^*_t)$. Each pair contains the bounded-context input and the expert action.

$$L_{\text{SFT}} = -\sum_{(x_t, a^*_t) \in \mathcal{D}_k} \log p_{\theta_0, \phi_k}(a^*_t \mid x_t)$$

The target is the executable action text, not a reasoning trace. The model learns to act from compact state, not from long prompt scaffolding.

### Stage 2: Refine with Rewards

SFT imitates experts, but online execution can wander into off-trajectory states. The refinement stage uses GRPO-style policy gradients with shaped rewards derived from the same tracker fields rendered into the state block.

$$r_t = r^{\text{env}}_t + r^{\text{prog}}_t - r^{\text{err}}_t$$

The progress term rewards advancing the current subgoal. The error term penalizes invalid, repetitive, or phase-inconsistent actions, plus a small step cost.

The reward rules are drafted offline with GPT-5.5 from the task specification, but they are fully deterministic at rollout—no LLM judge is queried during training.

A reference-KL penalty keeps the RL policy close to the SFT adapter, preventing format drift.

## What They Found

With Qwen3-8B, SFT+RL reaches 89.6% unseen success on ALFWorld, 76.8% success on WebShop, and 66.4% unseen success on SciWorld.

![Grouped bar chart comparing success rates across ALFWorld, WebShop, and SciWorld for SFT-only, SFT+RL, and baseline methods]({{ "/assets/figures/4bd73e808358/fig_007_02.png" | relative_url }})

These numbers are competitive with the strongest published baselines—1.2 points below GiGPO's 90.8% on ALFWorld, essentially tied with EMPO2's 76.9% on WebShop success rate, and ahead of EMPO2's 75.9 on SciWorld score at 79.7.

The context savings are stark. Prompt tokens per turn drop from 1.3k → 0.18k on ALFWorld, 3.1k → 0.49k on WebShop, and ~2.0k → 0.49k on SciWorld, each relative to full-history ReAct.

![Bar chart comparing prompt tokens per turn between full-history ReAct and constant-context method across three benchmarks, showing 4–7× reduction]({{ "/assets/figures/4bd73e808358/fig_008_03.png" | relative_url }})

Total token use per episode drops ~10–12× on ALFWorld, ~14× on WebShop, and ~2–3× on SciWorld.

SFT alone already carries most of the weight. On WebShop, the untrained prompt variant with a state block reaches only 23.6% success; SFT raises that to 62.2%.

RL then pushes it to 76.8%, adding 13–28 percentage points on ALFWorld, 15–17 on WebShop, and 5–15 on SciWorld across backbones.

The ablations confirm that the state block is the critical prompt-side signal. Remove it after training, and success drops 15.6 points.

The model uses the state block as execution state—it does not memorize trajectories.

Subgoal-shaped rewards add 4.6 points over terminal-only rewards.

All of this runs on a single A100 80GB GPU, compared to prior RL baselines that report using multiple H100s.

## The Catch

The approach is not yet plug-and-play. State trackers and reward rules are specified per task family.

The authors draft them offline with GPT-5.5, but the rules still require validation before RL training. Scaling to arbitrary new domains means writing new trackers and reward schemas for each one.

The evaluation covers three text-mode environments: ALFWorld, WebShop, and SciWorld. GUI and multimodal agents are acknowledged future work.

Each per-family LoRA module does not zero-shot transfer to unseen skills—you need to train a new adapter for each workflow.

But the modularity is a feature too. Adding a new skill requires only training a new ~0.5 GB adapter—the base model stays fixed, and existing skills are preserved without interference.

## What This Opens Up

This paper shows that procedural context in agents can be systematically moved from prompts into weights. The recipe is modular and lightweight: a deterministic tracker renders compact state, a LoRA adapter absorbs the reusable procedure, and inference runs on bounded context.

For personal agents navigating privacy, cost, and capability simultaneously, this is a practical path forward. A local 8B model with a 0.5 GB adapter and constant context can handle recurring workflows without sending growing histories to the cloud—while matching agents that burn tens of thousands of tokens per episode.

The re-reading ends here.
