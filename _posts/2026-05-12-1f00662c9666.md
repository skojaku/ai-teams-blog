*TL;DR: A single controller that both critiques draft answers and routes them to the right model closes most of the gap to the strongest agent — while calling that agent under 25% of the time.*

## Why Routing Collapses

You have a pool of heterogeneous LLMs. A smart router should spread work across them — some queries to the lightweight model, some to the heavyweight.

In practice, without explicit load-balancing, the router sends everything to the strongest model.

Every query. Every turn.

That's not coordination. That's abdication.

Fang, Yuan, Lan, Han, and Brinton ask: can a single controller both *critique* intermediate drafts and *route* queries to the right agent, iteratively, while actually balancing the load? Their answer is yes — and it closes most of the gap to the strongest agent while using it for under 25% of total calls.

## One-Shot Routing Doesn't Verify

Most multi-agent LLM systems treat coordination as a single decision: pick a model, get an answer, return it. No second looks.

RouterDC (Chen et al., 2024) learns a one-shot router by matching query embeddings to agent embeddings. It never revisits the answer.

Router-R1 (Zhang et al., 2025) adds RL and multi-turn interaction, but it targets retrieval — decomposing queries into sub-questions, each routed separately. There's no mechanism to critique a draft and decide whether it's good enough. The reward signal is outcome-level: it can't tell you which turn earned credit.

CoRL (Jin et al., 2025) conditions routing on externally specified budget modes. You tell it "low" or "high" budget up front. The controller doesn't autonomously decide when to stop, and it doesn't model iterative critique.

All three share the same blind spot: they route, but they don't *verify*.

## The Loop: Critique, Route, Refine

The authors cast multi-agent coordination as a finite-horizon MDP. At each turn, the controller does three things at once:

1. **Evaluate** the previous agent's draft (verification).
2. **Decide** whether to stop or continue (stopping).
3. **Select** the next agent if continuing (routing).

Turn 1 is pure routing — the controller sees the query and picks an agent. From turn 2 onward, the controller sees the current draft and outputs both a critique and a routing decision. The selected agent then receives the query, the previous draft, and the controller's critique.

![Architecture diagram showing the iterative critique-and-routing controller evaluating agent drafts and routing to heterogeneous agents over multiple turns]({{ "/assets/figures/1f00662c9666/fig_002_01.png" | relative_url }})

This dual role — critic and router — is the core mechanism. The controller isn't just a dispatcher. It's an editor: it reads each draft, flags problems, and sends the work to whichever agent is best positioned to revise.

## Walking Through the Reward

Each controller action $a_t$ encodes a routing decision and a verification judgment. The composite reward is a 50/50 mix:

$$r_t = 0.5 \, R_r(s_t, a_t) + 0.5 \, R_v(s_t, a_t)$$

$R_r$ is a binary routing reward — did the selected agent produce a correct answer?

$R_v$ is a binary verification reward — was the critique sound?

Boundary cases are zeroed out: no verification on turn 1, no routing on the final turn.

The training objective maximizes expected discounted return subject to per-agent utilization constraints:

$$\sum_{t=1}^{T} \mathbb{E}\left[\mathbf{1}(f(a_t) = k)\right] \leq \rho_k \sum_{k'=1}^{K} \sum_{t=1}^{T} \mathbb{E}\left[\mathbf{1}(f(a_t) = k')\right] \quad \forall\, k$$

$\rho_k$ caps agent $k$'s share of total invocations. The strongest agent gets $\rho_1 = 0.25$ — at most 25% of calls.

The constrained problem is relaxed via Lagrangian multipliers. The modified routing reward becomes:

$$R'_r(a_t, s_t) = R_r(s_t, a_t) - \sum_{k=1}^{K} 2\xi_k \cdot \mathbf{1}(f(a_t) = k)$$

Every time agent $k$ is selected, its routing reward gets penalized by $2\xi_k$. When agent $k$ is over-used, $\xi_k$ rises and the penalty discourages further selection. The full objective is then optimized with GRPO, the group-relative policy gradient method.

## What the Numbers Show

Two systems, seven math benchmarks. System 1 uses Qwen2.5-7B-Base as the controller with 30B, 7B, and 1.5B agents. System 2 uses Qwen3-4B-Base with 30B, 8B, and 1B agents.

**System 1**: 80.5% average accuracy. Router-R1 scores 69.1%. An 11.4 percentage point gap.

**System 2**: 82.3% average accuracy. Router-R1 scores 66.5%. A 15.8 pp gap.

![Bar chart comparing accuracy across seven math benchmarks for the proposed method versus baselines in both system configurations]({{ "/assets/figures/1f00662c9666/fig_006_02.png" | relative_url }})

The strongest agent alone — Qwen3-30B-A3B-Instruct — scores 85.9%. So the iterative controller closes most of the distance while invoking the heavyweight for under a quarter of calls.

### The Lagrangian Penalty Is Not Optional

Remove the Lagrange multipliers ($\xi_1 = \xi_2 = \xi_3 = 0$) and the system collapses. Every query gets routed to the strongest agent.

Under constrained evaluation — where any agent that exceeds its budget is cut off — accuracy drops from 80.5% to 56.5% in System 1, and from 82.3% to 50.1% in System 2.

The penalty is the difference between a functional system and a broken one.

### One Refinement Round Does Most of the Work

On MATH-lighteval, accuracy jumps sharply from 1 turn to 2 turns: 72.8% → 88.7% in System 1. Beyond 2–3 turns, returns saturate.

![Line plot showing accuracy versus number of reasoning turns, with sharp improvement from turn 1 to 2 and plateau thereafter]({{ "/assets/figures/1f00662c9666/fig_008_03.png" | relative_url }})

Most prompts don't exhaust the turn budget. The loop converges fast.

## Honest Limits

The experiments stay within math reasoning benchmarks, where correctness is binary and verification rewards are easy to define. Open-ended domains — creative writing, summarization, code review — would need learned or LLM-based judges instead of rule-based rewards.

Multi-turn interaction adds latency and compute. The paper acknowledges this trade-off but doesn't quantify it. If your application is latency-sensitive, the cost of 2–3 agent calls per query may not be worth the accuracy gain.

The utilization caps ($\rho_k$) and Lagrangian multipliers ($\xi_k$) are hand-chosen. The sensitivity analysis shows stability within a moderate range, but the principled selection of these hyperparameters for an entirely new domain is left open.

Iterative critique-and-routing — *critique, route, refine* — turns a fleet of unequal models into something stronger than any single one, without defaulting to the biggest.
