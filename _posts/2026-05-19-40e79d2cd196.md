*TL;DR: A new survey argues that the software wrapper around an LLM — tools, sandboxes, memory, validators — is the real bottleneck on agent reliability, and reframes that wrapper as "code as agent harness": the executable, inspectable, stateful medium through which agents actually get things done.*

You built the smartest model you could. It aces benchmarks, generates fluent code, reasons in chains.

Then you drop it into a long-horizon task — debugging a repo, navigating a GUI, running a lab experiment — and it falls apart.

Not because the model is dumb. Because the scaffolding around it can't hold.

This survey from Ning et al. gives that scaffolding a name: **code as agent harness**. The claim is that code is not just what agents produce — it is the runtime medium through which they reason, act, remember, verify, and coordinate. Taking that medium seriously is the central engineering challenge for reliable agentic AI.

## Why the Wrapper Matters More Than You Think

Most work treats code as a target artifact — the thing the model outputs, and you check whether it compiles. But in agentic systems, code plays at least three other roles simultaneously.

It is the **reasoning substrate**: when a model writes a Python function to solve a math problem and an interpreter runs it, the result is not just an answer — it is a verifiable trace of intermediate steps.

It is the **action interface**: when a robot or a GUI agent turns intent into executable policy code, the program becomes the contract between the model and the environment.

And it is the **environment model**: when an agent reads repository state, test results, and execution logs to decide what to do next, it is interacting with a world represented as code.

Three properties make code uniquely suited to this role. Code is **executable** — model outputs become operations with verifiable outcomes, not just text that sits there.

Code is **inspectable** — intermediate computation is exposed as structured traces that the harness can read, store, and act on.

Code is **stateful** — the evolving program represents task progress in a persistent, modifiable form across steps.

Pure natural-language chain-of-thought has none of these properties. You can't run a paragraph. You can't inspect its variable bindings. You can't branch on its runtime state.

## Three Layers of the Harness

The survey organizes the landscape into three connected layers — how code becomes operational inside the agent loop.

![Three-layer taxonomy of code as agent harness, spanning the harness interface, harness mechanisms, and scaling to multi-agent settings]({{ "/assets/figures/40e79d2cd196/fig_002_01.png" | relative_url }})

### Harness Interface: Code Connects the Model to the World

The first layer establishes what code *does* for the agent.

Code for reasoning externalizes internal logic into verifiable computation — the model proposes procedures, the harness executes them, observes runtime behavior, and feeds results back.

Voyager's persistent skill library in Minecraft showed that executable code artifacts can accumulate across tasks. Code as Policies showed that LLM-generated Python programs can serve as real robot-control policies. SWE-bench demonstrated that code repositories, tests, and traces constitute a structured environment where agents plan, act, and revise.

These are not unrelated ideas. The survey reframes all of them as instances of a single principle: code is the interface that makes reasoning executable, action programmable, and environment state inspectable.

### Keeping the Loop Running

Once code is inside the agent loop, the harness has to decide what to execute next, remember what happened, surface the right tools, and turn failures into repairs.

This is the second layer — the mechanisms that sustain a code-centric agent over long horizons.

Planning methods decompose tasks, ground plans in repository structure, or search over trajectories. Memory systems maintain working state, retrieve relevant code context, and store reusable experience. Tool use connects the agent to APIs, sandboxes, and verification environments.

And feedback-driven control closes the loop: static analysis catches bugs before execution, runtime errors feed back into repair, and test results become the verification signal for iterative debugging.

The key pattern across all of these mechanisms is the **plan-execute-verify cycle**. The agent proposes code, the harness runs it, verification produces a signal, and the agent revises.

This cycle is what makes a harness *adaptive* — not just a fixed pipeline, but a closed-loop system that converts failures into progress.

### Scaling to Many Agents

The third layer extends the harness from one agent to many.

When multiple agents operate over shared code, the harness must coordinate roles — manager, planner, coder, reviewer, tester — manage interaction modes — pair programming, repair, debate, red-teaming — and maintain consistent shared state across repository edits, test runs, and execution traces.

The survey positions shared code artifacts — repositories, test suites, traces, structured workflows — as the coordination substrate for multi-agent systems. The same properties that make code a good harness for a single agent (executable, inspectable, stateful) also make it a good medium for collective verification.

One agent writes a patch, another reviews it, a third runs the tests — all through the shared code artifact, all inspectable by the harness.

## Infrastructure vs. Artifacts: A Sharp Line

One of the sharpest conceptual moves in the paper is the distinction between **system-provided harness infrastructure** — the predefined tools, APIs, sandboxes, memory systems, validators, and permission boundaries that the system builder sets up — and **agent-initiated code artifacts** — the tests, skills, DSL programs, traces, and temporary tools that agents create, execute, observe, revise, persist, and share *during task execution*.

![Comparison of system-provided harness infrastructure versus agent-initiated code artifacts, showing how predefined tools differ from artifacts agents create and evolve within the task loop]({{ "/assets/figures/40e79d2cd196/fig_009_03.png" | relative_url }})

Voyager's expanding skill library is an agent-initiated artifact. A regression test that an agent writes to verify its own patch is an agent-initiated artifact. A temporary debugging script is an agent-initiated artifact.

The survey argues that agent-initiated artifacts are the underexplored frontier. We have spent years building better infrastructure. We have spent far fewer years studying what happens when agents start writing, running, and revising their own harness code — and the verification, regression, and coordination problems that creates.

## What the Survey Reveals

The paper's contribution is taxonomic and conceptual rather than empirical. It synthesizes methods from program-delegated reasoning through multi-agent PR workflows into a unified three-layer framework.

It identifies six categories of open challenge for harness engineering that go beyond model capability:

- **Evaluation adequacy** — current benchmarks measure final task success, not whether the harness supported reliable execution along the way.
- **Semantic verification** — execution feedback can confirm code runs, but not that it does the right thing.
- **Regression-free self-evolution** — when agents modify their own harness, how do you prevent cascading failures?
- **Transactional shared state** — multiple agents editing the same code need something like database transactions to avoid conflicts.
- **Human-in-the-loop safety** — safety-critical actions need oversight, and the harness must represent that oversight as inspectable state.
- **Multimodal extension** — the code-centric harness needs to integrate visual, auditory, and physical signals it cannot currently represent.

## The Honest Limits

As a survey, this paper proposes a conceptual framing and organizes existing work. It does not introduce new empirical results or a formal benchmark.

The "harness engineering" agenda is proposed but not yet validated — we do not know whether this taxonomy will survive contact with the next generation of agents.

The breadth of domains covered — robotics, GUI agents, scientific discovery, personalization — means domain-specific depth is limited. A robotics researcher will find the embodied-agent section thin; a software-engineering researcher will want more on repository-level harnessing.

The taxonomy may need refinement as any single area matures.

## Why This Reframing Changes How You Build

If you are building an agentic system, this survey invites you to stop thinking about code as output and start thinking about it as the medium through which your agent operates.

The question shifts from "can the model generate correct code?" to "does the harness make the agent's code executable, inspectable, and stateful enough to close the loop?"

That reframing has practical consequences.

You start caring about execution traces as first-class objects. You start treating test suites as environment models, not just quality gates.

You start designing multi-agent coordination around shared code artifacts rather than message-passing protocols. You start asking whether your agent can write and revise its own harness code — and what breaks when it does.

The harness was always there. We just weren't calling it by name.
