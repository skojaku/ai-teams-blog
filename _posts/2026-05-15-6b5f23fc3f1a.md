*TL;DR: You can replace the 60K-line deterministic backend of an agentic benchmark with an LLM-inferred proxy state and still get stable, model-differentiating evaluation with over 90% human agreement—a trade-off that collapses months of engineering into a scenario spec.*

## Sixty thousand lines of overhead

AppWorld's benchmark engine weighs in at roughly 60K lines of code.

Another 40K for the benchmark itself, plus about 1,800 unit tests—built over 14 months.

Meanwhile, production teams shipping tool-calling agents need evaluation rigs that evolve with the product, not ones that fossilize while the backend is still under construction.

The question sounds almost reckless: what if you throw out the deterministic database entirely and let an LLM infer the state instead?

The answer is workable. Proxy State-Based Evaluation keeps the principle that made AppWorld and τ-bench attractive—score the final outcome, not the trajectory—but replaces the heavy backend with LLM-inferred state and LLM judges.

You keep the rigor. You lose the engineering tax.

## Ditching the deterministic backend

The key insight from AppWorld and τ-bench was that many correct tool-calling paths solve the same user goal, so matching against a single ground-truth trajectory is wrong by design. Final state-based evaluation—check the database after the conversation ends—solves that.

But checking the database requires *having* a database. That's the 60K-line problem.

Proxy State-Based Evaluation sidesteps it with a scenario schema $z$ that encodes five slots: the user goal $g(z)$, user facts $u(z)$, system facts $s_0(z)$, the expected final state $s^*(z)$, and expected agent behavior $b^*(z)$.

No deterministic engine. No unit tests. Just a structured spec.

![Architecture diagram showing five LLM components in the proxy state-based evaluation loop: reasoning agent, user simulator, tool simulators, state tracker, and dual judges]({{ "/assets/figures/6b5f23fc3f1a/fig_014_04.png" | relative_url }})

Then five LLM components cooperate in a loop:

1. **Reasoning agent** — the model under test, running a ReAct-style loop.
2. **User simulator** — generates next utterances conditioned on the scenario's goal and user facts.
3. **Tool simulators** — respond to tool calls by "reading from" and "writing to" the proxy state.
4. **State tracker** — after every tool step, infers the current proxy state $\tilde{s}_{t,k} = f_{\text{state}}(\tau_{\leq(t,k)})$.
5. **Judges** — a goal-completion judge and a hallucination-detection judge score the final outcome.

The critical design choice: the state tracker conditions on the *full trajectory prefix* at every step, not just the previous state.

Step-by-step updating $\tilde{s}_{t,k} = f(\tilde{s}_{t,k-1}, \cdot)$ accumulates errors over long conversations. Full prefix conditioning avoids that trap.

## How the judges work

The LLM-as-judge idea got its validation from Zheng et al.'s MT-Bench and Chatbot Arena work, which showed that LLM judges can reliably rank open-ended model outputs. The same principle, applied to structured state-based verification.

Two judges run after the conversation ends.

**Goal-completion judge** — classifies the outcome as *completed*, *user-error*, or *agent-error* by comparing the proxy final state $\tilde{s}_T$ and the agent's last message against $s^*(z)$ and $b^*(z)$.

**Hallucination judge** — returns two binary flags: tool-side fabrication ($h_{\text{tool}}$) and user-side fabrication ($h_{\text{user}}$). Tool hallucination means the simulator produced information not in $s_0(z)$ or the subquery. User hallucination means the user simulator drifted from $u(z)$ or $g(z)$.

Human–LLM agreement on goal completion hits 82.7% three-way (both annotators + LLM judge). On hallucination detection, it climbs to 94.7%.

Those numbers come from 50 conversations independently annotated by two domain experts. Thin for a framework that aspires to replace deterministic backends—but directionally strong.

## What the benchmark reveals

The benchmark runs 208 scenarios across commerce and account management, split into 157 training and 51 testing scenarios. Every scenario is rolled out 10 times for robust estimates.

### Stronger models win, consistently

Goal completion scales monotonically with both model size and reasoning effort. GPT-5 at high reasoning effort scores 85.76%, medium drops to 80.53%, low to 66.04%, minimal to 59.83%.

![Goal completion rates across model families at varying reasoning effort levels, showing monotonic scaling with model strength and reasoning budget]({{ "/assets/figures/6b5f23fc3f1a/fig_005_01.png" | relative_url }})

The ordering holds across Gemini and Qwen families too—Gemini-2.5-Pro beats Flash, and Qwen3-235B beats 30B.

This monotonic scaling is the signal that the benchmark is actually differentiating capability, not noise.

### Training on proxy-state data transfers

Supervised fine-tuning on filtered successful trajectories pushes the Qwen3-30B-A3B base model from 65.64% to 77.34%—an 11.7 percentage point gain. Rejection-sampling fine-tuning (RFT) is more modest, reaching 67.11%.

The biggest SFT gains show up in long conversations. At 5+ turns, the base model collapses to 42.75%—after SFT it recovers to 69.44%.

![SFT improvement stratified by conversation turn depth, showing the largest gains on long conversations where the base model struggles most]({{ "/assets/figures/6b5f23fc3f1a/fig_006_03.png" | relative_url }})

That's a +26.69 pp jump on the hardest stratum. The proxy-state training signal generalizes to multi-step interactions where it matters most.

### Hallucination rates stay near zero

Under the default configuration (GPT-5o, medium reasoning), tool hallucination runs at 1.33% and user hallucination at 0.67%.

Swap in a weaker state tracker—GPT-4o instead of GPT-5o—and tool hallucination climbs to 3.61%. The proxy state degrades, and the tool simulators that read from it propagate the errors.

A manual audit of 50 intermediate state transitions confirms the state tracker: 92% correctness with 0.86 inter-annotator agreement (Fleiss' κ).

### Cross-model robustness holds

Switch all simulators and judges from GPT-5o to Gemini-2.5-Pro and the model rankings barely shift. SFT still improves the base model; GPT-5-family models still outperform Gemini-2.5-family models.

No single-family circular bias—a real concern when one LLM both generates and evaluates.

## The honest limits

Proxy state is *inferred*, not verified against ground truth. Its reliability rides entirely on the state-tracker LLM's quality and the completeness of the scenario specification.

Under-specify the system facts and tool hallucination rises. Under-specify the user facts and user hallucination rises. The ablations show this clearly: stripping 100% of system facts pushes tool hallucination to roughly 25%.

All 208 scenarios are synthetic and cover exactly two domains—commerce and account management. There is no evidence yet that this approach transfers to domains where state semantics are ambiguous or the "correct" final state is itself contested.

And the human agreement numbers, while strong, come from 50 conversations annotated by two experts. That is a thin validation sample for a framework that aspires to replace deterministic backends.

## What this opens up

The practical payoff is twofold. First, you can stand up an agentic benchmark quickly and evolve it with product roadmaps—write scenario specs, not database engines.

Second, the benchmark-as-environment design produces both evaluation metrics and training data in a single pipeline. Successful rollouts become on-policy SFT data; all rollouts become rejection-sampled RFT data.

For teams building tool-calling agents, this collapses two pipelines into one: you evaluate and you collect training data from the same runs.

Sixty thousand lines was the old price of rigor. The new one is a well-written spec.
