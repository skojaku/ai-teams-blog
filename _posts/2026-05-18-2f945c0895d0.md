*TL;DR: DMoA replaces fixed multi-agent org charts with a differentiable router that dynamically selects which agents participate at each reasoning step—achieving 89.38% average accuracy across 9 benchmarks while using fewer tokens than any self-evolving baseline.*

## No More Org Charts

Imagine a team of specialists where, at every step of a hard problem, a manager decides on the fly who actually needs to speak next.

Not everyone talks every round. Not everyone even shows up. The roster expands and contracts as the problem demands.

That is what DMoA does for LLM agents. Instead of wiring agents into a chain, a star, or a graph before the problem even starts, it routes agents step by step using a learned, differentiable mechanism—treating multi-agent collaboration the way a Mixture-of-Experts layer treats tokens.

## The Static Compilation Problem

Most multi-agent systems today are built on a design-then-compile pipeline. You pick a topology—chain, tree, star, graph—and every query runs through that same wiring.

Static templates like Chain or Star fix both the agents and the communication pattern. Self-evolving approaches like G-Designer and SafeSieve improve things by generating a query-specific graph, but they still *pre-compile*: the full workflow is determined before reasoning begins, then executed rigidly.

The original Mixture-of-Agents framework (Wang et al., 2025) takes a step toward flexibility. It concatenates every agent's output at each layer and feeds the result to every agent in the next layer.

But that is a dense, fully-connected design—every agent, every step. No sparse routing, no adaptive pruning.

What you want is the MoE trick Shazeer et al. (2017) taught us: out of many experts, route to only the few that matter. DMoA lifts that idea from tokens to reasoning steps.

## How the Router Works

DMoA starts with an agent pool $\mathcal{A} = \{a_1, a_2, \dots, a_N\}$, where each agent bundles an LLM, a profile, and a tool set.

Defining the pool costs nothing at runtime—an agent is only instantiated when the router selects it.

At each reasoning step $i$, the router answers two questions: *who* should participate, and *how many*?

![Architecture diagram showing the DMoA pipeline: a Sentence Transformer encodes context, an RNN-Router produces routing logits from the encoded context and hidden state, and a top-K-plus-threshold rule activates a sparse subset of agents per step, with entropy feedback for training]({{ "/assets/figures/2f945c0895d0/fig_005_01.png" | relative_url }})

### Encoding the context

A Sentence Transformer compresses the current context into a semantic vector. For the first step, that context is just the query:

$$X_1 = \text{SentenceTransformer}(P_{\text{usr}})$$

For later steps, the router summarizes what happened by taking a weighted sum of the routed agents' response embeddings. The weights come from the router's own logits, so the system learns which prior answers matter most:

$$X_i = \sum_{j=1}^{k_{i-1}} \alpha^{(j)}_{i-1} \, \hat{R}_{i-1,j}$$

### The RNN-Router

The core routing decision uses a GRU-based RNN that carries hidden state $h_i$ across steps. This is where history lives—the router remembers who it selected before and adjusts accordingly:

$$h_i, o_i = \text{RNN-Router}(X_i,\, h_{i-1}), \quad z_i = \text{Linear}(o_i)$$

The logits $z_i \in \mathbb{R}^N$ go through a top-K-plus-threshold rule: an agent is selected only if its normalized weight exceeds the uniform baseline $1/N$ *and* it ranks in the top $K$. This means the number of active agents per step $k_i$ is adaptive—sometimes two, sometimes five, never forced.

### Why entropy as the training signal?

You cannot backprop through an LLM's text output. RL-based methods like those used in GPTSwarm and G-Designer treat the whole system as a black box, collecting sparse outcome rewards over dozens of episodes.

DMoA takes a different route. After running *all* agents in a step during training, it computes each agent's predictive entropy—the average token-level conditional entropy over the response:

$$\text{PE} = -\frac{1}{M}\sum_{j=1}^{M}\sum_{x_j} P_\Theta(x_j \mid x_{<j}) \log P_\Theta(x_j \mid x_{<j})$$

Low entropy means high confidence. The router should prefer confident agents. So we convert negative entropy to a confidence score $C_i = \text{Softmax}(-E_i)$ and align the router logits with a pairwise ranking loss:

$$\mathcal{L}^{(i)}_{\text{rank}} = \sum_{a,b} \mathbb{1}\!\left(C^{(a)}_i > C^{(b)}_i\right) \log\!\left(1 + \exp\!\left(-(Z^{(a)}_i - Z^{(b)}_i)\right)\right)$$

This is self-supervised, dense, and it flows gradients cleanly through the RNN and the aggregation weights. No rollout, no sparse reward.

## The Numbers

Across 9 benchmarks, DMoA hits an average accuracy of **89.38**, versus 83.91 for SafeSieve—the best self-evolving baseline. A 5.47-point gap.

The biggest gains land on the hardest tasks. On DDXPlus (medical diagnosis), DMoA scores 83.37 versus Vanilla's 56.40—a +26.97 jump. On DS-1000 (data science code generation), it is +25.94. The router is clearly learning to call in specialists when the problem demands them.

And it does this while burning fewer tokens. On GSM8K, DMoA uses roughly 1.4 million tokens versus GPTSwarm's 3.3 million, while scoring 98.87% against 90.20%. Sparse activation is not just elegant—it is efficient.

![Token consumption comparison on GSM8K: DMoA reaches 98.87% accuracy with ~1.4M tokens, while GPTSwarm achieves only 90.20% at ~3.3M tokens, with other self-evolving baselines falling between]({{ "/assets/figures/2f945c0895d0/fig_009_03.png" | relative_url }})

### Test-time training

Because the entropy signal needs no labels, you can optimize the router on the fly. Feed DMoA the first 10–30 queries from a new benchmark, collect entropy, update the router, and keep going.

This test-time training variant roughly matches the few-shot version on its own. Combined, Few-shot + TTT pushes the five-benchmark average to roughly 89.45—and MMLU alone to 93.50.

### Open-source agents beating GPT 5.2

Swap in Qwen3-32B, Llama3-70B, DeepSeek-V3-67B, and Mistral 8×22B as the agent pool. DMoA with these open-source models outperforms GPT 5.2 and Gemini 3.2 Pro on most settings.

The swarm compensates for any single model's weaknesses.

### Robustness under attack

Under prompt-injection attacks, static multi-agent baselines drop 6.5–12.7% in accuracy. DMoA? Barely a dent—its adaptive routing mechanism preserves performance, with only a slight decrease before and after attacks.

## Where It Breaks

The agent pool is predefined. DMoA selects among existing agents with fixed capabilities; it cannot mint a new agent at runtime. If no agent in the pool excels at a task like formal theorem proving, the router picks the least-bad option.

The RNN-Router and Sentence Transformer add parameters that need 40–80 few-shot queries to optimize. Cheap by most standards, but not zero-shot.

The main results use a single backbone (gpt-oss-120b), so we do not yet know how the router behaves when the pool spans wildly different model sizes and architectures. The open-source ensemble results are encouraging, but the routing dynamics under architectural diversity remain an open question.

And step-wise routing means the system decides *who* participates per step, but each step is still a full agent invocation. A finer-grained router might route at the sub-task level—whether that pays off is another question entirely.

## The Bigger Picture

DMoA shows that the MoE concept does not stop at tokens. The same sparse-gating, confidence-driven routing that revolutionized model scaling can be lifted to the reasoning level, where the "experts" are entire agents with distinct models, profiles, and tools.

The result is a multi-agent system that is elastic in both space and time—expanding and contracting its roster as each problem unfolds.

No org chart required.
