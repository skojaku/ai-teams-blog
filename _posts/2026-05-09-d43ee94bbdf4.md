*TL;DR: A multi-agent workflow that decomposes competitive programming into five specialist roles—algorithm selection, knowledge retrieval, reasoning, coding, and error checking—boosts acceptance rates by 6.48% across models without changing a single parameter, while LoRA fine-tuning on the same data manages just 0.89%.*

## The Seven-Fold Gap

You can pour compute into training, scale your data, tune your LoRA ranks—and gain less than one percentage point on acceptance rate.

Or you can reorganize how the same model *thinks* about a problem and gain six.

That's the headline from MAS-Algorithm. Same backbone, same weights, no gradient updates. The difference is a structured pipeline that mirrors how competitive programmers actually work: pick an algorithm, pull relevant knowledge, reason through the solution, write the code, then check for errors and loop back if needed.

AlphaCode proved you could scale training data and model capacity to tackle competition-level code generation. MapCoder showed that multi-agent collaboration helps on standard benchmarks. But neither knits algorithm selection, knowledge retrieval, structured reasoning, implementation, and feedback-driven debugging into a single closed loop.

MAS-Algorithm does—and the numbers speak.

## Five Agents, One Pipeline

The framework decomposes algorithmic problem solving into five specialist agents, each with a clear job.

**Agent1 – Algorithm Selector.** Given a problem statement, Agent1 predicts relevant algorithm and data-structure labels from a predefined taxonomy. It's formulated as multiple-choice selection with rationales. Agent1 supports multi-branch output—multiple (label set, rationale) pairs—enabling parallel exploration of alternative solution paths.

**Agent2 – Domain Knowledge Provider.** Agent2 runs a retrieval-augmented generation pipeline over OI-WIKI, conditioned on the problem and Agent1's predicted labels. It retrieves conceptual knowledge—algorithm templates, standard techniques—and summarizes it for downstream use. The embedding model is bge-zh, and retrieval is top-$k$ with LLM-based aggregation.

**Agent3 – Logical Reasoner.** This is the core reasoning module. Agent3 takes the problem, selected algorithms, and retrieved knowledge, then produces a structured solution plan: problem analysis followed by numbered, actionable solve steps, optionally with pseudocode.

Agent3 has two modes—*reasoning* (initial plan) and *replanning* (revise a failed plan based on feedback)—intentionally decoupling planning from implementation.

**Agent4 – Code Implementer.** Agent3's plan becomes Agent4's input. Agent4 translates the structured plan into compilable C++. It supports two modes: *implementation* (fresh code) and *revision* (fix code based on feedback that doesn't require a strategy change).

**Agent5 – Error Checker.** Given the problem, the reasoning trace, the code, and execution results, Agent5 produces a structured feedback report with four parts: Analysis, Error Identification, Remediation Guidance, and a Control Signal.

That signal is one of three:

- **PASS** – solution is correct, stop.
- **FIX** – implementation bug, route back to Agent4.
- **RETHINK** – the strategy is fundamentally flawed, route back to Agent3.

This three-signal loop is the heart of the system. It differentiates local coding mistakes from global reasoning failures, directing feedback to the agent that can actually fix the problem.

## What Sits Between the Agents

The agents don't just pass raw text to each other. Three deterministic tools sit between them.

The **Format Conversion Tool** standardizes intermediate outputs into structured JSON.

The **Sample Extraction Tool** parses sample test cases from problem descriptions using regex—deliberately avoiding LLM-based extraction to prevent cascading errors.

The **Judging Tool** runs submitted code in a Docker container (gcc:latest), executes test cases, and returns detailed feedback: AC/WA/TLE status, pass counts, execution time, memory, and mismatched outputs for failed cases.

These tools make the pipeline reliable where it needs to be reliable. You don't want an LLM hallucinating your sample inputs.

## What the Numbers Say

Across five Qwen-series models, MAS-Algorithm delivers an average AC rate improvement of 6.48% over direct prompting, with gains ranging from 4.39% (Qwen3-235B) to 9.00% (Qwen3-14B).

The mid-scale model benefits most. Qwen3-14B sees +9.00% AC rate and +10.80% case pass rate—structured workflow guidance compensates for capacity limitations that larger models partially overcome on their own.

On LiveCodeBench-Pro (Q4 2024–Q2 2025), the framework generalizes with 4.72%–4.99% improvements across correctness metrics. This isn't an artifact of an in-domain benchmark.

Geometry and matrix problems see the largest category gain: +16.00% AC rate. Mathematically intensive tasks—number theory, dynamic programming—see smaller gains, suggesting that deeper formal reasoning remains the bottleneck.

### Workflow Beats Fine-Tuning

Here's the punchline. LoRA fine-tuning on the same data improves AC rate by just 0.89%. The workflow on the same model delivers 6.62%.

That's roughly a 7× gap. Parameter-efficient fine-tuning barely moves the needle. Inference-time orchestration moves it substantially.

And it gets worse for training. When the authors fine-tune Agents 1, 3, 4, and 5 jointly with LoRA, AC rate *drops* 15.03%. Multi-agent fine-tuning actually degrades performance—likely because the distilled supervision data isn't clean enough and the training signals for different agents conflict.

### The Upper Bound Is Far Away

The replacement experiments are the most revealing part of the paper. Rather than removing modules (which would break the pipeline), the authors swap in oracle or enhanced variants.

Replacing Agent3 (reasoning) with distilled CoT from ground-truth solutions yields +27.7% AC rate. Replacing Agent2 (knowledge retrieval) with an enriched knowledge base yields +22.95%. Even the error checker (Agent5) with oracle failure reports contributes +9.40%.

Agent1 (algorithm selection) with oracle labels? Barely moves the needle—−0.49%. The authors attribute this to the knowledge integration module: noisy but diverse label predictions might actually help Agent2 retrieve broader useful context, while strict ground-truth labels constrain it.

The takeaway is stark: reasoning is the biggest lever, knowledge is second, and the current implementations of both are far from their ceilings.

## The Honest Limits

Iterative refinement has diminishing returns. Solutions often pass sample tests but fail on full evaluation cases—local correctness doesn't guarantee hidden-case correctness.

Models *persist* in incorrect interpretations despite repeated RETHINK feedback. Some failure modes are not fixable through more rounds of prompting.

The error taxonomy drives this home. Core modeling errors, complexity misjudgment, and semantic misunderstanding account for over half of all failures. These are not bugs you can patch.

The multi-agent fine-tuning result is a warning, not a roadmap. Training individual agents within this workflow requires better data quality and standardized supervision signals that simply don't exist yet.

And while the framework generalizes across model scales and benchmarks, it's validated only on algorithmic programming. Whether the same decomposition helps in other reasoning-intensive domains remains an open question.

## Orchestration Over Parameters

The core insight isn't that agents are better than fine-tuning. It's that *how* you organize inference matters more than *what* you train. Five specialist roles, a closed feedback loop with three control signals, and deterministic tools between the gaps—together they extract far more from an unchanged model than LoRA does from a changed one.

The map now shows where the hills are.
