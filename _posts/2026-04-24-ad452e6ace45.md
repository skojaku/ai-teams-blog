*TL;DR: A multi-agent LLM system with shared memory generates tabular features that beat every baseline across 23 datasets—mean rank 1.12 on classification, 1.29 on regression—all for about $0.17 a dataset.*

## A $0.17 Feature Engineer

Seventeen cents. That is what MALMAS—the Memory-Augmented LLM-based Multi-Agent System—costs you per dataset, end to end.

On 16 classification benchmarks it achieves a mean rank of 1.12; the next-best method, OpenFE, sits at 3.12.

The gap is not close.

The shift here is structural, not incremental. Prior LLM-based feature generators like CAAFE and OCTree iterate over proposals but carry no structured memory of what worked or why—each round starts roughly from scratch.

MALMAS decomposes feature generation across six specialized agents, runs them for four rounds, and feeds every round's results into a three-tier memory that shapes the next round. The result is a search that learns what works and why—something a memoryless LLM loop simply cannot do.

## The Old Way Hits a Ceiling

Traditional automated feature engineering relies on operator libraries. DFS composes features from fixed aggregations; AutoFeat stacks log and polynomial transforms; OpenFE searches symbolic operators with boosting and pruning.

Fast and interpretable, but blind to task semantics—these methods can combine `age` and `income` because both are numeric, but they cannot reason that `income_per_capita = income / household_size` captures a signal that raw division misses.

LLM-based methods brought semantics to the table. CAAFE reads a task description and proposes transformations in natural language; OCTree pairs an LLM with evolutionary search over operator trees.

But both lack a mechanism that remembers what failed or adapts strategy across rounds. The exploration is disconnected from the learning signal—you get redundant features and no way to steer the next attempt.

## Six Agents, One Memory

MALMAS breaks the problem along three nearly orthogonal dimensions—transformation complexity, data scope, and data-type dependency—and assigns each dimension its own agent.

The six agents are:

- **Unary-Feature Agent**: single-column transforms (log, binning, scaling).
- **Cross-Compositional Agent**: multi-column interactions (ratios, differences, products).
- **Temporal-Feature Agent**: time-series patterns (lags, rolling windows).
- **Aggregation-Construct Agent**: group-level summaries (mean by category, count by bin).
- **Local-Transform Agent**: region-specific transforms (piecewise thresholds, localized scaling).
- **Local-Pattern Agent**: latent patterns via clustering or local interaction modeling.

A Router Agent reads the dataset metadata and the accumulated global memory, then activates only the agents that are relevant. No time column? The Temporal-Feature Agent sits out. That is not waste; that is focus.

## How Memory Works

Each agent carries three memory layers.

**Procedural memory** logs what was tried—base columns, transformation type, feature name, description, round index. It exists to suppress redundant exploration.

$$\text{ProcMem}_i^{(r)} = \{(b_j, t_j, f_j, d_j, r)\}_{j=1}^{n_i^{(r)}}$$

**Feedback memory** records what worked—each feature paired with its validation metric value and an effectiveness flag. This is explicit credit assignment.

$$\text{FeedMem}_i^{(r)} = \{(f_j, m, v_j, e_j, r)\}_{j=1}^{n_i^{(r)}}$$

**Conceptual memory** distills *why* it worked. After each round, the LLM compresses procedural and feedback traces into reusable heuristics—rules like "ratio features between monetary columns and group-level aggregations tend to be effective"—keeping the prompt context lean.

$$\text{ConMem}_i^{(r)} = \text{LLM}(\text{ProcMem}_i^{(r)},\ \text{FeedMem}_i^{(r)})$$

After all active agents finish a round, a Summary Agent aggregates their local feedback and conceptual memories into a **Global Conceptual Memory**. This shared prior conditions routing and prompting in the next round.

Agent A discovers that log-transformed monetary features are effective; Agent B learns that interaction features between those and categorical aggregations also help. The summary fuses both insights, and neither agent duplicates the other's search.

## Four Rounds, Twelve Features

MALMAS runs four rounds by default. At each round:

1. The Router activates a subset of agents based on metadata and global memory.
2. Each active agent constructs a prompt from dataset statistics, its own feedback and conceptual memories, and the global memory.
3. The agent calls the LLM (DeepSeek-V3 in the paper's main experiments) to propose transformations, instantiates them as code, and evaluates the resulting features with a downstream model (XGBoost).
4. It updates its three local memory layers.
5. The system selects the top-3 features per round—only the best survive—and appends them to the dataset.

$$\mathcal{X}^{(r+1)} = \mathcal{X}^{(r)} \cup \text{TopN-Features}(\mathcal{T}^{(r)},\ \mathcal{F}^{(r)})$$

Three features per round, twelve total. The memory ensures those twelve are informed by every round of feedback that preceded them.

## What They Found

The numbers are decisive. On 16 classification datasets, MALMAS achieves a mean rank of 1.12—OpenFE, the next-best, sits at 3.12.

On 7 regression datasets, MALMAS posts a mean NRMSE rank of 1.29; OpenFE again second at 2.86.

![Mean AUC rank across 16 classification datasets—MALMAS leads every baseline by a wide margin]({{ "/assets/figures/ad452e6ace45/fig_004_02.png" | relative_url }})

The ablation tells the origin story. Starting from a bare single-agent base, mean rank is 5.11—adding agents one by one brings it down to 2.30. Then the memory module stacks on top and collapses the rank to 1.12.

![Ablation from a single agent to the full system with memory: mean rank drops from 5.11 to 1.12]({{ "/assets/figures/ad452e6ace45/fig_007_03.png" | relative_url }})

Memory is doing more than diversification alone could ever achieve.

Crucially, the generated features generalize across downstream models. Tested with XGBoost, LightGBM, Random Forest, and MLP, MALMAS maintains a mean rank of 1.00—the next-best method trails at 2.40.

The features carry signal, not just XGBoost-specific quirks.

And the cost: 0.452 hours, 147.57k tokens, roughly $0.17 per dataset using DeepSeek-V3. A rounding error compared to the labor cost of manual feature engineering.

## Where It Runs Out of Road

MALMAS needs labeled data and a downstream evaluation signal every round. When labels are scarce or your evaluation budget is tight, the iterative feedback loop starves.

Each round retrains the downstream model on a growing feature set—more candidates mean more retraining, and that retraining becomes a bottleneck as the feature pool swells.

The framework is designed for tabular data only. Whether the multi-agent-plus-memory idea transfers to images, text, or graphs is an open question—not a promised next step.

And while MALMAS provides transformation descriptions and memory traces for interpretability, the LLM's generation process is not fully transparent. You can see *what* it tried and *whether* it worked, but the chain from prompt to proposal is opaque.

## The Takeaway

Feature generation has always been a search problem masquerading as an engineering problem. MALMAS turns the search iterative and gives the searchers shared notes. Six agents, three memory layers, four rounds, twelve features—and a mean rank that sits alone at the top.

The memory remembers so you do not have to.
