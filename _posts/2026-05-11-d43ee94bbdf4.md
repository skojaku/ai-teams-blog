*TL;DR: A five-agent pipeline that decomposes competitive programming into algorithm selection, knowledge retrieval, reasoning, coding, and error diagnosis beats LoRA fine-tuning by 7× on the same data — without touching a single model weight.*

## The Weight That Wasn't

You can spend weeks curating algorithmic training data, tuning LoRA adapters, and squeezing out a 0.89 percentage-point gain in acceptance rate.

Or you can hand the same model a structured workflow — pick the algorithm, fetch the knowledge, plan the solution, write the code, diagnose the errors — and watch acceptance climb by 6.48 percentage points on average. Same models. Same data. No weight updates.

That is the headline from MAS-Algorithm, and the gap is not subtle. The workflow approach delivers roughly seven times the gain of fine-tuning on the identical task distribution.

The old way: throw more compute at training and hope reasoning emerges. AlphaCode's approach — massive sampling and filtering — works, but at substantial cost and with limited interpretability. The new way: decompose the problem-solving process into modular stages, each handled by a specialized agent, and let the pipeline self-correct through structured feedback loops.

## How the Pipeline Works

Competitive programmers don't just read a problem and dump code. They classify the problem type, recall relevant algorithms, sketch a solution plan, implement it, then debug.

MAS-Algorithm mirrors this workflow across five agents:

**Agent1 — Algorithm Selector.** Takes the problem statement and predicts algorithm and data-structure labels. Formulated as a multi-choice selection over a predefined tag set. Can propose multiple branches, which flow through the pipeline in parallel — because hard problems often have more than one viable approach.

**Agent2 — Domain Knowledge Provider.** A RAG pipeline over OI-WIKI. The predicted tags become queries; the retrieved articles get summarized into concise reference material for downstream reasoning.

**Agent3 — Logical Reasoner.** The core of the system. Given the problem, tags, and retrieved knowledge, it outputs a structured solution plan: problem analysis followed by numbered solve steps, sometimes with pseudocode. It operates in two modes — a *reasoning mode* for initial plans and a *replanning mode* for when things break.

**Agent4 — Code Implementer.** Translates the plan into C++. Also dual-mode: fresh implementation or targeted revision.

**Agent5 — Error Checker.** The feedback engine. It consumes the problem, the reasoning plan, the generated code, and execution results, then produces a structured error report with one of three control signals:

- `PASS` — done, terminate
- `FIX` — local implementation bug, route back to Agent4
- `RETHINK` — fundamental strategy error, route back to Agent3

![Pipeline diagram showing the five-agent MAS-Algorithm architecture with feedback loops from the Error Checker back to the Logical Reasoner and Code Implementer]({{ "/assets/figures/d43ee94bbdf4/fig_003_01.png" | relative_url }})

That `FIX` vs. `RETHINK` distinction is where the self-correction loop gets its teeth.

A wrong loop bound goes to Agent4 for a surgical fix. A flawed algorithm choice goes all the way back to Agent3 for a fresh plan. No more treating every error as a code bug when half the time the *strategy* is wrong.

## What the Numbers Say

Across five Qwen models (8B through 235B-scale), MAS-Algorithm lifts the average AC rate by 6.48 percentage points. The mid-scale Qwen3-14B benefits most — a 9.00% jump, with case-pass-rate gains exceeding 10%.

![Bar chart comparing AC rates between Direct prompting and MAS-Algorithm across five Qwen model sizes, showing consistent improvements with MAS-Algorithm]({{ "/assets/figures/d43ee94bbdf4/fig_005_02.png" | relative_url }})

Meanwhile, LoRA fine-tuning Qwen3-Coder-30B on the *same* dataset yields a 0.89% AC rate improvement. That is roughly one-seventh of the workflow gain, achieved at non-trivial training cost.

And here is the real kicker: when the authors fine-tuned *all* the pipeline agents simultaneously, the 30B model's AC rate *dropped* by 15.03 percentage points compared to the non-fine-tuned pipeline. Multi-agent fine-tuning, at least with current data quality, does not help — it actively hurts.

On the harder LiveCodeBench-Pro benchmark, MAS-Algorithm nearly doubles performance: from 5.28% to 10.00% AC rate on Qwen3-Coder-30B.

## Where the Headroom Lives

The most revealing experiments are not the baseline comparisons — they are the replacement studies.

The authors swap individual agents with oracle or enhanced variants to probe how much room remains.

Replacing Agent3 (the Logical Reasoner) with distilled chain-of-thought from ground-truth solutions yields a staggering **+27.7%** AC rate. The reasoning stage is the bottleneck, by a wide margin.

Replacing Agent2 (the Knowledge Provider) with an enhanced knowledge base that includes curated problem-solution pairs gives **+22.95%**. Better knowledge retrieval directly translates to better reasoning.

Replacing Agent1 (the Algorithm Selector) with ground-truth labels? The AC rate actually *drops* by 0.49%, while case pass rate inches up by 0.08%. Near-zero impact either way. The current RAG system apparently does not cleanly exploit precise labels — noisy-but-diverse predictions sometimes help more than they hurt.

![Replacement and ablation study results showing Agent3 distilled CoT yielding the largest improvement at +27.7% AC rate]({{ "/assets/figures/d43ee94bbdf4/fig_008_04.png" | relative_url }})

These numbers tell you exactly where to invest: reasoning quality first, knowledge base second, label accuracy a distant third.

## Where the Loop Snaps

The framework is not equally effective everywhere. Math and number theory problems improve by only 3.62% AC rate. Dynamic programming by 4.83%.

Geometry and matrix problems jump by 16%.

The pattern: tasks requiring deep formal reasoning and precise mathematical modeling see smaller gains. The models simply lack the underlying capability, and no amount of workflow scaffolding fully compensates.

The iterative loop also converges fast and then plateaus. Solutions that pass sample tests but fail hidden test cases are a persistent problem. And models sometimes *persist* in wrong interpretations despite explicit feedback, pointing to a hard ceiling: some failure modes need stronger base reasoning, not more rounds of prompting.

Prior multi-agent systems like MapCoder demonstrated that multi-agent code generation could beat single-LLM approaches on competitive programming. Earlier frameworks tackled pieces of the pipeline — simulation-driven debugging, for instance — without integrating algorithm selection, knowledge retrieval, and structured feedback into one coherent workflow. MAS-Algorithm's contribution is precisely that integration: a single pipeline from problem understanding through error diagnosis, with the FIX/RETHINK feedback loop closing the gap between implementation bugs and strategy failures.

## Why This Matters

This paper makes a clean empirical case: for algorithmic reasoning, inference-time orchestration is a far more effective lever than training-time updates.

Not just slightly better — seven times better on the same data distribution.

The replacement paradigm also gives researchers a principled diagnostic tool. Instead of asking "does this component help?" you ask "how much would the *ideal* version of this component help?" The answer — +27.7% for reasoning — is a much more useful signal for deciding where to allocate research effort.

The honest limit: both the agent designs and the tools are preliminary. The gap between current performance and the oracle upper bounds is wide, especially in reasoning and knowledge. Multi-agent fine-tuning degrades rather than improves, likely due to noisy distilled data. And the pipeline still cannot rescue a model that fundamentally misunderstands a problem.

Workflow decomposition amplifies capability. It does not create it from nothing.

Seven times the gain, zero weight changes — the leverage is in the loop.
