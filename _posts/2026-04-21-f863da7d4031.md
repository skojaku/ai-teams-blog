*TL;DR: ExTAGENTS is a multi-agent framework that solves the problem of feeding LLMs more knowledge than fits in their context window — and it actually gets better with more input, unlike prior multi-agent methods that paradoxically degrade past 128k tokens.*

## The Paradox at the Wall

Feed a multi-agent LLM system more knowledge than fits in one context window and it doesn't get better. It gets worse.

LLM×MapReduce, the prior state of the art, "shows no advantage over directly truncating context" beyond 128k tokens. More information, worse answers.

ExTAGENTS flips that curve. Feed it 1,024k tokens and it consistently outperforms baselines that top out at 128k.

## Two Bottlenecks, Not One Problem

The failure comes down to two bottlenecks.

**Synchronization bandwidth.** When agents split a massive input across themselves, they need to share what they've found. Chain of Agents and LongAgent restrict each agent to a bandwidth of 2 — you only hear from your immediate neighbors. LLM×MapReduce expands this to $O(L/|m|)$ where $|m|$ is the expected message length, but the large budgets encourage agents to forward dense context chunks, which drowns signal in noise.

**Reasoning overload.** The reasoning agent that synthesizes the final answer typically stuffs as much information as possible into its context. The ratio of redundant information swamps the useful bits, and performance collapses as input scales.

Both problems are about information flow — too little in synchronization, too much (and too repetitive) in reasoning.

## How Every Agent Sees Everything

ExTAGENTS gives every seeking agent global visibility.

At each timestep $t$, agent $a_{i,t}$ sees the full set of peer messages $M_{t-1} = \{m_{j,t-1}\}_{j=1}^{N}$. But seeing everything would blow through the context window.

The fix: each agent *rates* the relevance of every peer message to the query, then keeps only the top-$k$ most pertinent ones:

$$\text{Top}_k(M_t) = \arg\max_{\hat{M}_t \subseteq M_t, |\hat{M}_t|=k} \sum_{j' \in \{j \mid m_{j,t} \in \hat{M}_t\}} h_{j',t}$$

The constraint is simple: $|q| + |d_i| + |\text{Top}_k(M_{t-1})| < L$. Pack as many relevant messages as the window allows, no more.

This turns the bandwidth from a fixed small number to effectively $N$ — every agent draws on the most relevant findings from the entire team, not just the agent sitting next to it.

## How the Reasoning Agent Accumulates

The reasoning agent doesn't try to ingest everything at once. Instead, it accumulates.

At iteration $s$, it pulls in the top-$2^s$ messages from the most recent synchronization round:

$$M_r^{(s)} = \text{Top}_{2^s}\left(\{m_{i,t^\star}\}_{i=1}^{N}\right)$$

The mechanism doubles the intake at each step — 2 messages, then 4, then 8.

At each step, the reasoning agent checks whether it can already answer the query. If yes, it stops early.

No redundant flood — the doubling lets the agent decide when it has enough, rather than drowning in accumulated noise.

## What the Numbers Show

On ∞Bench+ (En.QA), ExTAGENTS hits F1 of .382 at 1,024k tokens, compared to LLM×MapReduce's .374 at 128k.

On HotpotQA with gpt-4o-mini, ExTAGENTS reaches F1 .534 at 1,024k input versus DRAG's .482 at 128k. With gpt-4o, ExTAGENTS pushes to F1 .597 at 1,024k — up from .553 for a single ExTAGENTS agent at 128k.

Every baseline plateaus or degrades past its context window. ExTAGENTS keeps climbing.

![Scaling F1 scores across input lengths: ExTAGENTS trends upward through 1,024k tokens while baselines flatten or decline beyond 128k]({{ "/assets/figures/f863da7d4031/fig_003_02.png" | relative_url }})

For long survey generation, ExTAGENTS scores 7.63 on LLM-as-a-Judge versus AutoSurvey's 6.75, with 191 citations (up from 113) and a lower duplication rate (1.80 vs 2.41).

## Both Parts Pull Their Weight

The ablation study is sharp: removing knowledge-accumulating reasoning "leads to a significant drop in performance, especially as the amount of external knowledge increases." Removing global knowledge synchronization also hurts, across input lengths.

Both components matter. But KAR is the one that prevents the system from drowning as input scales — without it, more knowledge actively hurts.

## Small Models Seek, Big Models Reason

Because seeking and reasoning are decoupled, you can assign different models to each role.

Pairing a 3B seeking agent with an 8B reasoning agent yields a ~3.1× speedup over using the 8B model for everything, while the F1 (.363) stays far closer to the 8B-only baseline (.383) than the 3B-only floor (.328).

Small models seek; big models reason. The speedup is real and the accuracy cost is modest.

## The Honest Limits

ExTAGENTS uses fixed-size chunking with no semantic boundary optimization. The authors state plainly that it "makes no attempt to optimize those boundaries." Semantic segmentation or overlap windows could help, but introduce coordination overhead.

The ranking mechanism depends on model quality. Weaker LLMs or out-of-domain inputs can produce unreliable relevance scores — the authors concede that "the rankings could be unreliable in some extreme situations."

Stronger models benefit more from ExTAGENTS, likely because they rank better. A virtuous cycle, but one that means weaker models see less improvement.

## ∞Bench+: A Harder Test

Along the way, the authors fix a real problem in long-context evaluation. Existing ∞Bench samples let models answer many questions by sweeping an 8k window — no long-range reasoning needed.

ExTAGENTS filters those out and adds back original ∞Bench samples longer than 128k tokens, producing ∞Bench+ with 294 English and 184 Chinese samples that genuinely require multi-hop reasoning over long spans. It's a benchmark that can't be shortcut.

More knowledge, better answers — that's the curve we always wanted.
