*TL;DR: OccuBench uses LLMs themselves as simulated work environments—Language World Models—so we can finally test agents on emergency triage, nuclear safety monitoring, customs processing, and 97 other professional tasks that have no public APIs or testbeds. The result: no single model dominates across industries, and agents silently fail when data degrades without error signals.*

## The Untestable Majority

Your API just returned truncated data. No error code. No 500. No timeout. Just fewer fields than you asked for, and a list that stops halfway.

Do you even notice?

Most agents don't. And that is the most unsettling finding from OccuBench.

## The Domains We Cannot Reach

Emergency department triage. Nuclear reactor safety monitoring. Customs import processing. Greenhouse irrigation control. These are precisely the domains where autonomous agents would deliver the most value—and precisely the domains where no benchmark exists.

WebArena tests web browsing. SWE-bench tests code repositories. OSWorld tests desktop operations. Between them, they cover a tiny fraction of professional work.

The vast majority of high-stakes occupational tasks—healthcare, energy, logistics, finance—are locked behind enterprise systems with no public APIs and irreversible real-world consequences.

OccuBench changes the geometry of what we can test. Instead of building a real environment for each domain, it uses an LLM *as* the environment.

## How the Environment Becomes a Prompt

The key insight: the environment itself can be simulated by a language model. Given a configuration $c = (\text{system prompt}, \text{tool schema}, \text{initial state}, \text{state description})$, an LLM becomes a stateful, interactive environment—a Language World Model (LWM).

Formally, at each step:

$$(s_{t+1}, o_{t+1}) = f_\theta(s_t, a_t; c)$$

The agent issues a tool call $a_t$. The LWM generates a structured JSON observation $o_{t+1}$, maintaining state through its context window rather than an explicit database.

![Architecture diagram showing the Language World Model evaluation loop: agent tool calls flow into the LWM, which generates JSON observations conditioned on configuration and conversation history, and trajectories are scored by a rubric-based verifier.]({{ "/assets/figures/9b1077aaf82e/fig_012_08.png" | relative_url }})

This transforms environment construction from an engineering problem into a configuration problem. Building a new WebArena domain requires deploying entire web applications.

Adding a domain to OccuBench requires specifying four text components.

The idea has antecedents. Generative Agents (Park et al., 2023) used LLMs to simulate social behavior. WebWorld (Xiao et al., 2026) trained an LLM-based web simulator for agent training. OccuBench carves its own niche: using LLM simulation for *evaluation*, not training, across 65 professional domains rather than web tasks.

## What the Benchmark Covers

OccuBench covers 100 professional task scenarios across 10 industry categories and 65 specialized domains, with 382 evaluation instances.

Each scenario maps to a real human job role—emergency triage nurse, operations engineer, customs officer, production scheduler—not an abstract task.

A multi-agent synthesis pipeline powered by Gemini-3-Flash-Preview generates environment configurations, task instructions, tool definitions, solution plans, and verification rubrics. Every instance is guaranteed solvable (verified by execution), discriminative (calibrated difficulty), and diverse (grounded in 16 non-overlapping sub-topics per scenario, each with a professional reference document).

Instances that are trivially easy or provably unsolvable get filtered out.

The final dataset averages 5.5 tools and 16.2 tool calls per task. These are not single-shot function calls. They are multi-step workflows in stateful environments.

## Testing on Two Fronts

OccuBench evaluates agents along two dimensions.

**Task completion** across 10 industries: agriculture, business, commerce, education, healthcare, industrial, public service, science, technology, and transportation. This reveals each model's occupational capability profile—not just whether it can do professional work, but *what kind*.

**Environmental robustness** under controlled fault injection. All tasks are synthesized in clean environments (E0). At evaluation time, faults are injected by appending rules to the LWM's system prompt:

- **E1 (Explicit faults):** HTTP 500 errors, timeouts, connection refused. The agent knows the call failed. The correct behavior is to retry.
- **E2 (Implicit faults):** Truncated data, missing fields, incomplete lists, stale cached values. The response looks superficially correct. The correct behavior is to detect the degradation and re-query.
- **E3 (Mixed):** Approximately half explicit, half implicit faults.

Faults are transient (retrying recovers normal results) and parameterized by count and duration.

## Silent Failures Are Deadlier Than Loud Ones

Average agent performance under implicit faults (E2: 53.4%) is *lower* than under explicit faults (E1: 62.6%). Pure implicit faults are harder than mixed faults (E3: 54.4%).

This is counterintuitive. You would think that mixing in explicit errors would make things harder, not easier. But explicit errors come with a clear signal—the call failed, try again.

Implicit degradation has no signal. The data *looks* fine. The agent must independently notice that something is wrong.

![Bar chart comparing agent completion rates under three fault injection conditions across 9 models, with E2 (implicit faults) consistently showing the lowest performance.]({{ "/assets/figures/9b1077aaf82e/fig_010_04.png" | relative_url }})

This matters enormously for deployment. Real APIs don't always crash loudly. They return partial data, drop fields, serve stale caches.

OccuBench shows that current agents are fundamentally worse at detecting silent corruption than overt failure—a 14.1-point gap between clean performance (67.5%) and implicit-fault performance (53.4%).

## No Single Model Rules Them All

The 15-model evaluation reveals that every frontier model has blind spots that single-domain benchmarks hide.

GPT-5.2 leads overall at 79.6%, but its Commerce score (67%) is 14 points below Qwen 3.5 Plus (81%).

Gemini 3.1 Pro tops Education (84%) and Science (81%) but struggles in Healthcare (62%).

Claude Opus 4.6 excels in Transportation (77%) yet trails in Commerce (53%).

![Grouped bar chart showing completion rates across 10 industry categories for the top 6 models, with each model displaying a distinct capability profile.]({{ "/assets/figures/9b1077aaf82e/fig_008_02.png" | relative_url }})

These aren't random fluctuations. Each model has a distinct occupational capability profile—different shapes on the radar chart, different strengths, different weaknesses.

An organization choosing an agent based on aggregate rankings would miss this entirely.

Open-source models are surprisingly competitive. Qwen 3.5 Plus (69.9%) and DeepSeek V3.2 (69.6%) outperform most Claude variants. The assumption that closed-source models uniformly dominate professional tasks does not hold here.

## Reasoning Effort Pays Off—Massively

For models with adjustable reasoning effort, the trend is unambiguous.

GPT-5.2 jumps 27.5 points from minimal reasoning (`none`: 54.7%) to maximum (`xhigh`: 82.2%).

Claude Opus 4.6 shows a similar pattern: its highest effort level (`max`: 73.8%) outperforms `low` (70.2%) by 3.6 points.

Deeper reasoning directly translates to better professional task execution. That 27.5-point climb is one of the clearest scaling results in the agent evaluation literature.

## Strong Agents Make Weak Simulators

The cross-simulator experiments surface a critical asymmetry: being good at the task is not the same as being good at simulating the environment.

GPT-5.2 ranks first as an agent (79.6%) but produces the worst simulation quality. Under the GPT-5.2 simulator, all agents average only 29.3%. Under Gemini Flash, they average 67.9%.

The failure modes are instructive. GPT-5.2 invents rooms that don't exist in the hospital environment (state fabrication). It drops critical agents from roster queries (entity omission). It enforces return-window constraints that aren't in the task specification (rule invention).

In every case, the agent's strategy is correct. It fails because the simulator violated the environment contract.

With a capable simulator, rankings are reliable: Gemini Flash and Qwen 3.5 Plus agree on 85.7% of pairwise model orderings, with the top-3 agents matching exactly. The lesson is practical: verify your simulator before trusting your evaluation.

## The Honest Limits

OccuBench evaluates decision-making processes, not numerical precision. The LWM understands that a drug interaction check should return contraindication warnings, but the specific values it returns are generated, not retrieved from a real database.

For domains where cent-level financial accuracy matters, LWM-based testing should complement—not replace—real-environment validation.

Simulator dependence is real. Tasks verified as solvable under Gemini-3-Flash may become unsolvable under a different LWM. The simulator is part of the evaluation apparatus, not a neutral observer.

## What This Unlocks

OccuBench makes any professional domain an LLM can understand into a testable environment. That is a qualitative shift.

Before this, extending a benchmark to a new industry required deploying real infrastructure. Now it requires writing a system prompt and a tool schema.

The finding that implicit data degradation is the hardest fault type for current agents has direct implications for anyone deploying agents in production. Your system will encounter silent corruption. Most agents won't notice.

No single model is safe across every industry—but every model has a domain where it surprises you. The question is no longer "which model is best?" It's "which model is best at *your* job?"
