*TL;DR: TreeMem converts a single coarse final reward into role-specific credit for each agent in a builder–summarizer–retriever memory pipeline, using tree-structured rollouts and Monte Carlo branch averaging — no handcrafted intermediate rewards required.*

## The Paycheck Problem

Picture a construction site where the builder, the editor, and the file clerk all receive the same bonus check at the end of the month.

If the building stands, everyone gets rewarded — even if the clerk filed nothing and the editor left errors on the table.

That is exactly how today's multi-agent memory systems are trained: the builder writes memories, the summarizer compresses them, and the retriever answers a question, then *all three* get the same binary 0/1 reward from the final answer's accuracy.

The builder never learns whether its memories were actually useful; the retriever never learns whether it fished the right ones out.

TreeMem, from Mao et al., replaces that shared paycheck with per-agent credit — derived automatically from the final reward, without any handcrafted intermediate signals.

## Why Uniform Rewards Fail

Memory-R1 (Yan et al., 2025) trains memory agents with a single QA-correctness reward applied uniformly to every role.

The signal is sparse and ambiguous.

A bad builder action gets rewarded if the retriever somehow salvages the right answer; a great builder action gets penalized if the retriever chokes.

Methods like CoMAM and Mem-α try to fix this by designing role-specific reward functions — evidence coverage for the builder, memory quality for the summarizer, retrieval precision for the retriever.

But those proxy rewards require costly annotations and risk *reward hacking*: agents game the proxy metric instead of actually improving downstream performance.

TreeMem sidesteps both problems. No uniform reward, no handcrafted proxies.

It estimates each agent's contribution by branching the pipeline into a tree and averaging over the branches.

## Growing the Tree

The standard memory pipeline is linear:

$$a_1 \sim \pi^1_\theta(\cdot | H), \quad a_2 \sim \pi^2_\theta(\cdot | a_1), \quad a_3 \sim \pi^3_\theta(\cdot | q, a_1, a_2)$$

Builder produces memories $a_1$ from history $H$, summarizer compresses them into $a_2$, retriever answers the query $q$.

TreeMem turns this line into a tree during training.

The builder samples $G$ candidate memories. For each builder action $a^i_1$, the summarizer branches $J$ times. For each builder–summarizer pair, the retriever branches $K$ times.

That yields $G \cdot J \cdot K$ complete trajectories — each one a full path from raw history to final answer, but rooted at different choices along the way.

![Tree-structured rollout showing the builder, summarizer, and retriever branching into multiple trajectories with leaf rewards propagated upward via Monte Carlo averaging]({{ "/assets/figures/acfff1ce1f00/fig_002_01.png" | relative_url }})

Only the leaves carry a reward: $R_{i,j,k} = F(p_{i,j,k}, p^*)$, where $F$ is a final-answer evaluator like QA accuracy.

## Credit by Branch Averaging

Now the key move: propagate rewards *upward* through the tree via Monte Carlo averaging.

The retriever gets its own leaf reward directly:

$$\hat{Q}^3_{i,j,k} = R_{i,j,k}$$

The summarizer's credit averages over all retriever branches that depend on it:

$$\hat{Q}^2_{i,j} = \frac{1}{K}\sum_{k=1}^{K} R_{i,j,k}$$

The builder's credit averages over every summarizer-and-retriever branch below it, plus a length penalty for verbatim copying:

$$\hat{Q}^1_i = \frac{1}{JK}\sum_{j=1}^{J}\sum_{k=1}^{K} R_{i,j,k} \;+\; \frac{|a^i_1|}{|H|}$$

That length penalty $|a^i_1|/|H|$ discourages the builder from dumping the entire conversation into memory — a cheap but useless strategy.

Each estimate asks the same question from a different height in the tree: "When this agent made this choice, how did downstream outcomes vary?"

The tree structure is a training-time device. At inference, the three specialized policies run as a standard single-pass pipeline — no branching needed.

## From Credit to Specialization

TreeMem then assembles a GRPO-style trajectory group by sampling one path from each of the $G$ subtrees, giving $G$ trajectories total.

For each agent $n$, the $G$ selected credits are normalized into group-relative advantages:

$$\hat{A}^n_i = \frac{\hat{Q}^n_i - \text{mean}_r\;\hat{Q}^n_r}{\text{std}_r\;\hat{Q}^n_r + \epsilon}$$

Each policy is updated with a clipped surrogate objective — standard GRPO machinery, but the advantage for agent $n$ is computed from *its own* subtree-estimated credit, not from the shared final reward.

This is what makes heterogeneous specialization possible: the builder, summarizer, and retriever all see different training signals derived from the same tree.

## What They Found

On PersonaMem with a Qwen-2.5 backbone, TreeMem hits 72% average accuracy across 32K/128K/1M token settings (0.69, 0.75, 0.71), compared to 67% for CoMAM — the strongest RL baseline.

That is a +7.5% relative improvement.

With Llama, TreeMem reaches 70% average versus CoMAM's 66% — a +5.5% gain.

![Accuracy comparison across methods on PersonaMem at three context lengths for Qwen and Llama backbones]({{ "/assets/figures/acfff1ce1f00/fig_003_02.png" | relative_url }})

The pattern holds across benchmarks. On LongMemEval, TreeMem scores 63.46 F1, beating Memory-R1's 54.36. On LOCOMO, it posts 48.88 F1, topping the prior best (GAM) at 45.77.

### Every agent needs tree credit

The ablation is sharp. Remove tree-based credit from all agents ("w/o tree") and Qwen 128K accuracy drops from 0.75 to 0.64.

Remove it from just the summarizer and Qwen 128K falls to 0.62. Remove it from just the builder and you get 0.65 — better than removing it from everyone, but still well below full TreeMem's 0.75.

Each agent's specialization degrades when it receives the wrong signal. The gap is consistent across both backbones and all three context lengths.

![Ablation results showing accuracy drops when tree-based credit is removed from individual agents versus all agents]({{ "/assets/figures/acfff1ce1f00/fig_008_03.png" | relative_url }})

TreeMem also converges in fewer training steps than both final-reward and task-specific-reward baselines.

Per-step wall-clock overhead stays marginal despite the branching — the policy update uses only $G$ sampled trajectories, not the full tree.

## The Honest Limits

The tree rollout requires $G \cdot J \cdot K$ forward passes per training step. That adds only modest wall-clock overhead per step, since the policy update itself uses just $G$ trajectories — but the rollout cost does scale with the branch factors.

Marginal returns diminish past small branch sizes, and the paper does not test systems with more than three agents.

All experiments use backbones up to 8B parameters (Llama-3.1-8B-Instruct for the retriever).

Whether the same branching recipe scales to larger models, deeper pipelines, or a dozen heterogeneous agents remains an open question.

## The Takeaway

TreeMem takes one of the thorniest problems in multi-agent RL — who deserves credit for the final outcome — and solves it by simply *trying more branches* and averaging.

No annotations, no handcrafted proxies, no reward hacking.

Just a tree, some averaging, and the final reward you already had.
