*TL;DR: An attacker who inserts ~60 adversarial tokens into a function description can force an LLM agent to call whichever function the attacker chooses — up to 100% of the time — and the attack generalizes across queries, domains, and even reasoning models.*

You type "create a private repo called Hello World" and the agent quietly deletes the repository instead.

That's not a prompt-injection trick or a misaligned response.

The model selected the right-looking tokens, filled in valid arguments, and called `delete_repository` — because someone poisoned the description of that function with a string of optimized gibberish.

The function name and parameters were legitimate. The intent was hijacked.

## Where the Attack Lives

Agentic AI systems increasingly rely on the Model Context Protocol (MCP) to expose tools to LLMs. An MCP server publishes a list of functions — names, parameter schemas, and natural-language descriptions.

The LLM reads those descriptions to decide which tool to call.

Those descriptions are just text. Anyone who can edit an MCP server's function description — a malicious package maintainer, a compromised dependency, a rogue plugin author — controls what the model "sees" about that tool.

The source code stays hidden. The description is wide open.

Prior work attacked function-calling agents through prompt injection (forcing the model to generate harmful content) or semantic preference (biasing the model toward one function over a similar one). But those approaches break when the available functions span diverse domains — a calculator next to a calendar next to a cloud API.

There's no semantic preference to exploit when the target function has nothing to do with the user's intent.

FHA doesn't care what the functions mean. It hijacks the *selection mechanism itself*.

## How the Hijack Works

FHA adapts the GCG adversarial-suffix attack from jailbreaking to the function-calling task.

In classic GCG, you optimize a token string that, when appended to a prompt, makes the model produce harmful output. FHA changes the target: instead of generating toxic text, the adversarial suffix forces the model to output a specific function name as its tool call.

The threat model is narrow and realistic. The attacker modifies *only* the natural-language description of a target function within an MCP server.

No access to the user prompt. No access to other functions. No code changes.

In the MCP setting, the implementation isn't exposed to the user — but the textual description is, and evaluating natural language is hard.

The optimization minimizes cross-entropy loss over the target function name:

$$L_{\text{adv}}(\hat{x}_{1:s}) = -\log\left[P_{\pi_\phi}(\hat{y}_{\text{fh}} \mid \hat{x}_{1:s})\right]$$

where $\hat{x}_{1:s}$ is the perturbed input (function descriptions + query), and $\hat{y}_{\text{fh}}$ is the target tool call. Crucially, FHA optimizes only for the *function name* — the model's own autoregressive process fills in the arguments afterward.

That makes the target sequence shorter and the attack more efficient.

For reasoning models like Qwen-3, which emit a chain-of-thought before calling a tool, FHA forces the model to skip thinking and jump straight to the hijacked function call. The target string includes the thinking-end tokens, so the model produces `</think>` then the attacker's function name — no reasoning step required.

## The Numbers

The numbers are stark. Across five models — Llama-3.2-3B-Instruct, Mistral-7B-Instruct-v0.3, and Qwen-3 at 1.7B, 8B, and 14B — FHA achieves:

- **97–100% Function-Name ASR** on instructed models (Llama-3B, Mistral-7B)
- **70–82% ASR** on reasoning models (Qwen-3 variants), which require suppressing the thinking step first
- **65–92% Slot-Filling ASR** — meaning most hijacked calls have valid parameters and could actually execute

![Function-Name Attack Success Rate across five models, with instructed models near 100% and reasoning models between 70–82%]({{ "/assets/figures/7e4253d1ad84/fig_007_01.png" | relative_url }})

The adversarial suffix needs only about 60 tokens hidden inside a function description.

That's roughly 5% of the total function-calling context, compared to 25–50% for classic GCG jailbreaking on AdvBench.

Scale the suffix from 10 to 60 tokens and ASR climbs from ~65% to ~97% on Llama-3B.

![ASR increasing from approximately 65% at 10 tokens to 97% at 60 tokens for Llama-3B, showing the relationship between suffix length and attack effectiveness]({{ "/assets/figures/7e4253d1ad84/fig_009_03.png" | relative_url }})

### The Baselines Barely Register

The closest prior attack, MPMA (MCP Preference Manipulation Attack), edits function names or descriptions to create textual preference — biasing the model toward one function over a similar one.

On the BFCL_v3_multiple benchmark, where functions span diverse domains, MPMA achieves 0–10% ASR. FHA achieves 70–100% across those same models.

The gap is structural. MPMA relies on semantic similarity between the prompt and the function description.

When `delete_repository` sits next to `get_weather`, there's no semantic preference to exploit. FHA is agnostic to semantics — it doesn't need the target function to resemble the user's intent.

![FHA achieving 70–100% ASR across models compared to MPMA's 0–10%, with function injection baselines also far below FHA]({{ "/assets/figures/7e4253d1ad84/fig_021_07.png" | relative_url }})

### One Suffix, Many Queries

FHA can be trained over batches of diverse queries and payload configurations, producing a single adversarial suffix that hijacks across multiple user intents:

- **Formulation-diversity batches** (same intent, rephrased): 88% direct ASR
- **Argument-variation batches** (same function, different arguments): 79% direct ASR
- **Cross-batch transfer**: 74–80% ASR when the suffix trained on one batch is tested on another

Attacks trained on a single intent don't transfer to unrelated intents — which gives attackers fine-grained control.

Want to hijack only queries about file operations? Train on that.

Want to hijack everything? Train on multiple intents.

### Surviving Codebase Changes

A poisoned tool description needs to keep working even as developers add, remove, or reorder functions. Universal FHA trained on batches varying function position and count maintains **>0.85 ASR even with 25 additional out-of-distribution noise functions** added to the payload.

The attack persists through the kinds of routine codebase changes that happen every day.

## Why This Matters

Every MCP-based agentic system deployed today — customer-support bots, coding assistants, cloud-management agents — exposes function descriptions to the model.

Those descriptions are a soft target that no one is guarding.

A hijacked tool call isn't a glitch in the output. It's a structurally valid API invocation with real consequences: deleting repositories, transferring funds, exfiltrating data.

The slot-filling numbers — up to 92% — mean these aren't toy attacks. The model fills in correct parameters most of the time.

And the attack scales in a way that prior approaches don't. MPMA works when all functions do roughly the same thing. FHA works when they don't.

## The Honest Limits

The evaluation covers models up to 14B parameters with at most 4 functions per payload. Production systems often run larger models with more available tools, and the effectiveness of FHA in those settings remains untested.

The adversarial optimization requires white-box access to compute gradients — you need the model weights. Cross-model transfer, a known property of GCG-style methods, is not tested here.

The suffixes look like gibberish to a human reader. A careful audit of function descriptions would catch them.

But MCP servers can expose dozens or hundreds of tools, and descriptions change regularly. Manual auditing doesn't scale.

A few dozen unreadable tokens in a function description, and the agent does someone else's bidding — silently, with valid arguments, and at scale.
