*TL;DR: Most production LLM agent failures aren't model failures — they're architecture failures at the seam where stochastic LLM outputs become deterministic system actions, and this paper names that seam, catalogs six runtime patterns around it, and gives you a five-step methodology for choosing which ones you need.*

## A 90-Percent Discount That Shouldn't Have Shipped

A workflow lands in the wrong state because an event handler reacted to a stale prompt.

A 90-percent discount reaches a customer because no policy gate sat between the LLM's proposal and the write.

A long-running renewal process loses its place because nobody decided whether the source of truth was an event log or a versioned row.

None of these are model defects. All of them are architectural choices the team made before the LLM ever ran inference. This paper by Vasundra Srinivasan gives that seam a name — the *stochastic-deterministic boundary* — and builds a full selection methodology around it.

## The Seam You Already Have but Haven't Named

The stochastic-deterministic boundary, or SDB, is a four-part contract: a **proposer** (the LLM), a **verifier** (a deterministic check on the proposal), a **commit step** (the durable write after acceptance), and a **reject signal** (the typed response back when verification fails).

![The Stochastic-Deterministic Boundary as a vertical seam separating the LLM's stochastic output from deterministic system actions, with verifier-and-commit logic at the boundary]({{ "/assets/figures/67c38f668942/fig_006_01.png" | relative_url }})

Every production agent runtime instantiates these four parts somewhere. Most teams just haven't written it down.

The audit makes this concrete. Srinivasan examines five widely-used open-source frameworks — openai/swarm, AutoGPT, LangChain Agents, CrewAI, and Microsoft AutoGen — and finds explicit verifier-and-commit logic at 19 of 21 LLM-to-action call sites.

The boundary is already there. What varies is how explicitly it's designed and how strictly each part is enforced. The range runs from a one-line JSON parse with no schema check in openai/swarm's `core.py` to a multi-stage pydantic-plus-LLM-as-judge auto-review-and-revise loop in MetaGPT's `ActionNode`.

The post-mortem data is even sharper. Across 21 published agent failure reports, 15 (71.4%) localize to weaknesses at the boundary itself. And 17 (81%) of the documented fixes added or strengthened verification, commit semantics, or reject signaling.

Two cases anchor the claim.

Promptfoo reports a customer's agent dropping 23 points in prompt-injection resistance — from 94% to 71% — when upgraded from GPT-4o to GPT-4.1 on an identical harness. The fix was strengthening the verifier at the boundary.

An openai-agents-js bug had rejected tool calls reported back to the model as `status: 'completed'`, causing the model to hallucinate success. The fix switched the reject signal to `status: 'incomplete'`.

Both cases show that the verifier is model-version-dependent and the reject signal is load-bearing, not peripheral.

## Three Concerns, Six Patterns

Around the SDB, the paper organizes three orthogonal concerns that every production runtime must answer: **Coordination** (how work splits and combines), **State** (how the system remembers across pauses), and **Control** (who decides what runs, and when to stop).

Each concern has two patterns in the current catalog, each traced to a specific distributed-systems result.

![Taxonomy of six runtime patterns organized across three orthogonal concerns — Coordination, State, and Control — with each pattern's distributed-systems lineage shown]({{ "/assets/figures/67c38f668942/fig_008_02.png" | relative_url }})

- **P1 — Hierarchical Delegation** (Coordination). One orchestrator owns the work and dispatches sub-tasks to specialists. The merge step belongs in deterministic code. The LLM proposes. Deterministic code decides. Hewitt's actor model and Wu et al. provide the lineage.

- **P2 — Scatter-Gather plus Saga** (Coordination). A coordinator fans out to symmetric peers; each logs a compensating action so that if peer B fails after peer C wrote to billing, we can undo it. García-Molina and Salem's sagas provide the compensation semantics — compensations run in reverse order. When compensation logic grows larger than the original action, the fix is to split the action, not to grow the saga.

- **P3 — Event-Driven Sequencing** (State). An append-only log is the source of truth. Consumers subscribe, react, and emit new events. Kreps's log and Helland's work on distributed transactions are the lineage.

  The LLM-specific wrinkle: the log stays deterministic, but LLM-based consumers interpreting it do not. Which brings us to a failure mode worth its own name.

- **P5 — Shared State Machine** (State). A durable versioned row is the source of truth. Workers are stateless and pure. They propose next state via compare-and-swap (CAS). The store rejects stale writes. `human_required` is a state, not a missing event. Corbett's Spanner and van der Aalst's workflow nets are the lineage.

- **P4 — Supervisor plus Gate** (Control). Two mechanisms side by side: supervision restarts what dies (borrowed from Erlang's OTP), and the gate refuses out-of-policy writes before they reach an external system. The gate is a deterministic rule check — not a model call. If your gate decisions take longer than the action they gate, your gate is running a model. That's a bug, not an architecture.

- **P6 — Human in the Loop** (Control). Four control planes: kill switch, escalation, approval, throttling. All four emit to a single audit trail. You don't need all four at v1. You do need to record which you're deferring and why.

The catalog is explicitly open and current, not complete. New patterns enter by answering three questions: which concern it answers, what failure mode it prevents that no existing pattern prevents, and what its typed-contract specification looks like.

## Why Momentum Beats Variance

Here's the structural argument that ties everything together. Model the long-run reliability of an agent system as:

$$y(t) = \mu t + \sigma \cdot \xi(t)$$

where $y(t)$ is observed reliability, $\mu$ is the architectural momentum (the reliability slope set by pattern choice and SDB strength), $\sigma$ is per-call variance amplitude from the stochastic proposer, and $\xi(t)$ is mean-zero noise.

The paper presents this as a metaphor, not a derived model — but the structural point is precise. The linear term $\mu t$ grows with calendar time. The noise term $\sigma \cdot \xi(t)$ is bounded. As agents spend more time in production, the trajectory shaped by architecture dominates the per-call noise.

Base-model improvements compress $\sigma$ with each generation. They do not, on their own, change $\mu$. As $\sigma$ shrinks, $\mu$ becomes the dominant lever.

This reframes the engineering conversation from "which model?" to "which boundary and which spine?"

The paper also names a failure mode that the SDB makes legible: **replay divergence**. The same input event, replayed on a newer model version, produces different downstream events than the first run produced. The log itself remains deterministic. The LLM-based consumers reading the log do not. Replay divergence is the diagnostic trigger for migrating from P3 (Event-Driven Sequencing) to P5 (Shared State Machine).

## Choose Your Patterns in Five Steps

The methodology produces a six-line architecture decision record — a written artifact that reviewers and auditors can read instead of guessing. Each line comes from one step.

**Step 1 — Classify the runtime.** How long does one unit of work last, and does the world change during that window? Conversational (seconds, user waiting), Autonomous (minutes, unattended task), or Long-Horizon (hours to days, world changes mid-flight). The class determines which concern dominates: Coordination for Conversational, Control for Autonomous with side-effects, State for Long-Horizon.

**Step 2 — Choose the spine.** The spine is what the system remembers between failures. Select P5 (Shared State Machine) as the spine if all three predicates fire: pauses longer than one hour, state at any pause not reconstructible from the original input, and the world changes during pauses. If only the first predicate fails, P3 (Event-Driven Sequencing) is sufficient. If the second predicate fails, you don't need a durable spine — reconstruct on demand. This is the most expensive choice to migrate later, so a senior engineer signs off in writing before you proceed.

**Step 3 — Wrap with coordination.** P1 if there's a single clear owner, sub-tasks are independent, and a deterministic merge is feasible. P2 if peers hit external systems with side-effects, some will fail and the rest must still produce useful results, or the cost of an inconsistent partial write exceeds the cost of a compensation log. You can use both — P1 at the outer layer, P2 inside a sub-agent.

**Step 4 — Bound with control.** Always include P4 (Supervisor plus Gate) if the workflow has any side-effects on external systems. Add P6 (Human in the Loop) if a wrong action is legally or financially consequential, cases fall outside the policy envelope, or auditors will ask who decided.

**Step 5 — Sequence the build.** Build the dashboard before the agent. The trace is the contract. Suggested v1 order: state schema and observability lens, gate and audit log, orchestrator and one sub-agent, remaining sub-agents, then P6 control planes. Deferring the first two past v1 is the most expensive kind of technical debt.

## Different Workloads, Different Architectures

The methodology produces meaningfully different architectures for different workloads, and that's the point.

A Conversational billing assistant needs only P1 + P4 — no durable spine, because the session is short, state is reconstructible, and the world doesn't change mid-call. The expert on the line is already the human-in-the-loop.

An Autonomous fall-out scanner needs P3 + P2 + P4 + light P6 — event-driven because the order pipeline already emits authoritative events, scatter-gather because source systems fail independently, a gate to refuse malformed pre-staged writes, and throttling to protect downstream.

A Long-Horizon contract renewal needs P5 + P1 + P2 + P4 + P6 — all three spine predicates fire, billing writes need saga compensation, and contract mergers need human escalation.

Two Long-Horizon workloads in the same class compose the same pattern set but differ in the details that matter. The contract renewal's saga covers billing writes from a single orchestrator. The number port-in's saga covers multi-carrier rollback where the donating carrier can reject after the receiving carrier has provisioned. Same patterns, different compensation surfaces.

The companion repository provides a runnable reference implementation against the IBM Telco Customer Churn dataset, exercising all six patterns in composition across 100 renewal scenarios.

The failure-signature catalog is immediately practical. Each entry maps an observable symptom in production logs to a likely cause and a corrective action. "Gate decisions take longer than the action they gate" — your gate is a model call. Move it to deterministic code. "The same event replayed on a newer model version produces different downstream events" — that's replay divergence, and it's the migration trigger from P3 to P5.

## The Honest Limits

The catalog is explicitly open. Six patterns, three concerns, and the paper acknowledges new patterns will emerge — limiting predictive coverage for workload classes the authors haven't seen yet. The validation is via worked examples and a reference implementation, not a large-scale empirical study. The 21 post-mortems in the failure classification are a small sample relative to the space of production failures. And the reliability model ($y(t) = \mu t + \sigma \xi(t)$) is presented as a metaphor, not a derived model — the structural insight holds, but don't treat the equation as a quantitative predictor.

These aren't weaknesses the paper hides. They're boundaries of the current framework that future work and community use will expand.

Name the boundary, choose the spine, build the dashboard first.
