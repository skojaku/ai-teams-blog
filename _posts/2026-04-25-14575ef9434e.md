*TL;DR: AI agents burn hours stuck in sequential API-call chains, but a new framework borrows the "speculate-then-verify" trick from CPU design and LLM inference to let faster models predict likely next actions and execute them in parallel — yielding up to 20% latency reductions with zero correctness loss.*

## A Chess Game That Takes an Hour

Two state-of-the-art agents sit down to play chess.

An hour later, the game is still running. Not because the moves are brilliant — because each move is an API call, and every call blocks until it returns.

The agent on move six stares at the ceiling while the agent on move five thinks.

The bottleneck isn't compute. It's waiting.

This paper asks a deceptively simple question: must agents interact with their environment in strict sequence? Their answer is no — and they have the speedups to prove it.

## The Old Way and the New Way

Today's agent loop is pure serialization: observe, call an API, wait, observe again, call another API, wait again. Every step blocks on the previous one.

The pattern is identical to the instruction-level parallelism problem that CPU architects solved decades ago with speculative execution.

The old way: sequential `call -> wait -> call -> wait`.

The new way: while the slow Actor is still thinking, a fast Speculator guesses the most likely next actions and pre-launches them in parallel. If the guess is right, the work is already done. If it's wrong, you throw it away and proceed normally.

Either way, correctness is never compromised.

This is the same speculate-verify pattern that Leviathan et al. introduced for LLM token generation in speculative decoding — but lifted from individual tokens up to entire API calls, tool invocations, and even human responses. Hua et al. (2024) explored depth-oriented multi-step lookahead planning for agents, verified by a stronger model. Guan et al. (2025) extended that with RL to dynamically choose speculation depth. This paper flips the lens: instead of going deep along one branch, it goes broad — launching *k* parallel single-step guesses and studying which strategy wins.

## How Speculation Runs

The framework introduces two roles into the agent loop:

- **The Actor**: the authoritative, slow executor — a frontier LLM, an external API, a human typing a response. Its outputs are ground truth.
- **The Speculator**: a cheap, fast model that predicts what the Actor will decide and what the next API call should be.

At each step, both run simultaneously.

The Speculator fires off *k* guesses for the next action, each triggering a speculative pre-launched API call.

These calls are cached. When the Actor finally returns the real answer, the system checks: does any guess match?

If yes, the cache hit eliminates the wait for the next step entirely. If no, you fall back to sequential execution.

![Architecture showing the speculative actions framework: the Actor and Speculator run in parallel, the Speculator predicts k next API calls that are pre-launched and cached, then verified against the Actor's actual output to commit or discard.]({{ "/assets/figures/14575ef9434e/fig_002_01.png" | relative_url }})

The key invariant is **losslessness**. The final trajectory is always identical to what a purely sequential agent would produce. Three mechanisms enforce this:

1. **Semantic guards**: the Actor confirms state equivalence before committing a speculative branch.
2. **Safety envelopes**: only idempotent or reversible actions are pre-launched — web searches, read-only API calls, sandboxed OS operations.
3. **Repair paths**: when a guess is wrong, rollback or compensating actions restore the correct state.

The whole algorithm fits cleanly into the MDP formulation that agents already use. Each step is an API call $(h, q)$ — target and parameters. The policy $\pi$ maps state to call; the transition $f$ advances the state. Speculation adds a prediction function $\hat{g}$ and a cache $C$ that maps $(h, q)$ to pending futures.

### Walking through the math

Under the assumption that each speculative branch implies the correct next call with probability *p* (independently across steps), and the Speculator latency is $\text{Exp}(\alpha)$ while the Actor latency is $\text{Exp}(\beta)$ with $\beta < \alpha$, Proposition 1 gives a clean closed form.

With *k* branches, the probability of at least one hit is:

$$p(k) = 1 - (1-p)^k$$

Each hit saves an expected $\frac{\alpha}{\beta(\alpha + \beta)}$ time units — the overlap between the fast guess and the slow real call.

The expected runtime ratio converges to:

$$\frac{\mathbb{E}[T_s]}{\mathbb{E}[T_{seq}]} \;\to\; 1 - \frac{p(k)}{1 + p(k)} \cdot \frac{\alpha}{\alpha + \beta}$$

The theoretical ceiling for single-step speculation is a 50% speedup — hit when prediction is perfect ($p = 1$) and the Speculator is infinitely fast ($\alpha \to \infty$). Depth-focused multi-step speculation can push past this ceiling, but the single-step version is strikingly simple to deploy.

## The Numbers

**Chess.** Using GPT-5 as both Actor (high reasoning effort) and Speculator (low reasoning effort, prediction-optimized prompt), 3-prediction speculation achieves 54.7% move prediction accuracy and 19.5% average time savings across 30-step games.

More predictions improve both metrics monotonically: 1 prediction saves ~11.8%, 2 save ~15%, 3 save ~19.5%.

![Speedup versus next-action prediction accuracy for different numbers of speculative branches, with the theoretical 50% upper bound annotated and empirical results reaching ~20% speedup at ~55% accuracy.]({{ "/assets/figures/14575ef9434e/fig_006_02.png" | relative_url }})

**E-commerce.** On τ-bench retail tasks, multi-model Speculator ensembles (GPT-5-nano paired with Gemini-2.5-flash) predict between 22% and 38% of API calls correctly.

The cheap models speculate in 2–3 seconds — well under the ~30 seconds of average user typing time. That means roughly one third of customer turns get an instant response.

**Multi-hop web search (HotpotQA).** Top-3 speculation predicts ground-truth API calls with 46% accuracy under strict matching — meaning the exact same search query and parameters. Even approximate matches would push this higher.

**OS hyperparameter tuning (lossy extension).** This is where it gets interesting.

The lossless requirement is relaxed: a fast Speculator applies provisional parameter tweaks every second, while the Actor deliberates for 10–15 seconds before overwriting with its final decision. The joint Actor–Speculator system converges to the optimal configuration in ~13 seconds at 0.17 cents, versus ~200 seconds and 2.18 cents for Actor-only.

Both latency and cost drop simultaneously — 15× faster convergence, 12× cheaper.

## Counting the Cost

More speculative branches means better hit rates but more tokens spent. Theorem 3 gives the cost increase ratio in closed form, governed by the same $p(k)$ as the speedup. Practitioners can tune *k* offline: plug in an estimated hit rate, read off the latency savings and the price tag, and choose an operating point.

But static *k* wastes budget on low-confidence steps. Theorem 4 shows that when you have per-branch confidence estimates, the optimal policy collapses to a greedy rule: add branches in descending confidence order while the marginal hit probability times a continuation value exceeds the marginal cost.

In the stationary case, this reduces to a single threshold $\Delta^*$ that you estimate once and apply at every step. Implementation is O(*k*) per step.

The chess experiments confirm it: a simple 50% confidence threshold achieves the lowest additional token cost while providing latency reduction comparable to always launching three branches.

## Honest Limits

The framework requires that speculative API calls be reversible or idempotent (Assumption 2). Many real-world operations — placing an order, deleting a database record, sending an email — don't satisfy this without additional sandboxing. The authors are upfront about this boundary.

Single-step speculation is theoretically capped at 50% speedup. Practical results with current models top out around 20%. Prediction accuracy varies significantly by domain: 55% in chess but only 22–38% in e-commerce. The speedup you get is the speedup your Speculator's accuracy earns.

And the framework only speculates one step ahead in its base form. Multi-step speculation can push past the 50% ceiling, but the analysis gets more complex and the cost grows with the number of speculative steps that accumulate before the Actor catches up.

## The Bigger Picture

This framework is general and environment-agnostic — it unifies LLM calls, tool APIs, MCP servers, and human-in-the-loop interactions under a single lossless abstraction. The insight is that every agentic step is an API call, and every API call is a chance to prefetch work.

The closed-form cost–latency tradeoff is the practical payoff. You don't need to guess or sweep hyperparameters — you can compute the right number of speculative branches for your domain, your model costs, and your latency budget.

And in domains where losslessness isn't required, the OS results show that speculation can reduce both latency and cost at the same time, which is somewhat remarkable.

The pattern scales. As agents get more capable and their environments get richer, the sequential bottleneck only grows. Speculative actions turn that bottleneck into an opportunity.

The agents were staring at the ceiling. Now they're getting ahead.
