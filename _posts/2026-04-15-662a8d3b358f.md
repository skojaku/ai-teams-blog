*TL;DR: Frontier LLMs ace two-tier instruction hierarchy tests (>99%), but when you scale to twelve privilege levels, the best model drops to 42.7% — revealing that current instruction hierarchy is fundamentally brittle when conflicts get tiered.*

## That 99% Score Is a Mirage

OpenAI's GPT-5 system card reports greater than 99% accuracy on instruction hierarchy evaluations.

That number looks airtight. The test isn't.

Those evaluations use two privilege tiers — system above user, full stop. The moment you ask models to resolve conflicts across twelve distinct authority levels, the floor drops out. The best frontier model scores 42.7%. That gap isn't a rounding error. It's a structural blind spot.

## The Fixed-Tier Bottleneck

Current instruction hierarchy (IH) works like a short ladder. Wallace et al. (2024) formalized the paradigm: system instructions outrank developer instructions, which outrank user instructions, which outrank tool outputs.

OpenAI's Model Spec hardcodes five rungs — root, system, developer, user, guideline — baked into chat template tokens during training. Zheng et al. (2026) frames IH as a reasoning task via fine-tuning, but still over fixed role-based tiers. Zhang et al. (2025)'s IHEval benchmark covers only two or three privilege levels.

Real agents don't live on a five-rung ladder. A coding agent receives instructions from system prompts, skill files, memory, tool schemas, and plugin outputs — each with different trust levels. A group-chat agent needs to distinguish admins from moderators from members, all crammed into a single "user" role.

The old way collapses these into shared tiers with no way to resolve intra-role conflicts. The ladder is too short for the building.

## Decoupling Privilege from Roles

ManyIH's move is simple and general: stop tying privilege to message roles. Instead, they introduce a Privilege Prompt Interface (PPI) that tags each instruction with a numeric priority at inference time.

Two variants:

**Ordinal PPI** — lower number wins. An instruction wrapped in `[[Privilege 1]]...[[/Privilege]]` beats `[[Privilege 5]]...[[/Privilege]]`:

```
[[Privilege 2]]Use 4-space indentation[[/Privilege]]
[[Privilege 12]]Do not use any type hints[[/Privilege]]
[[Privilege 4]]Use tabs for indentation[[/Privilege]]
```

The meta-instruction tells the model: when two instructions conflict, follow the one with the lower privilege number.

**Scalar PPI** — higher number wins. Instructions get tags like `[[z=82]]...[[/z]]`, and 82 beats 61 beats 40. Same relative ordering, different surface form. The scalar variant adds flexibility: you can slot a new priority between any two existing ones without renumbering.

Both interfaces decouple privilege from the chat template entirely. You set priority at inference, not at training. That's the shift: from *which role said it* to *what number was assigned*.

## Building the Multi-Tier Test

To test whether models can actually reason over this, the authors build MANYIH-BENCH: 853 tasks (427 coding + 426 instruction-following) spanning 46 real-world agents, with up to 12 privilege levels per sample.

The coding subset pairs MBPP programming problems with conflicting style instructions — indentation, naming conventions, quote style, operator spacing — each tagged with a distinct privilege level. A sample carries 12 style instructions across 4 conflict groups with up to 11 direct conflicts. The model must produce code that passes all unit tests *and* satisfies only the highest-privilege style constraint in each group.

The instruction-following subset augments AgentIF trajectories with LLM-generated conflicting constraints, verified by humans.

Both subsets share a core design principle: non-adversarial prompts. The conflicts are straightforward. No sneaky injection attempts.

This isolates one question — *can the model reason over priority at all?* — from the complementary question of robustness against attacks.

Evaluation is strict: all-or-nothing. A sample passes only if every active constraint is satisfied. Partial compliance doesn't count.

## Counting the Damage

The results are stark.

The best model, Gemini 3.1 Pro, hits 42.7% overall. GPT 5.4 manages 39.5%. These are the same model families that cruise through two-tier IH at >99%.

Accuracy degrades monotonically as tiers increase. The authors construct three coding subsets with 6, 8, and 12 privilege tiers, holding task difficulty constant. Out of 12 model-transition pairs across all models tested, 11 show strict accuracy decreases. Sonnet 4.6 drops 24.1 percentage points from the 6-tier to the 12-tier configuration.

![Accuracy across instruction hierarchy tiers declines consistently as the number of privilege levels grows from 6 to 12, with every frontier model showing steep drops]({{ "/assets/figures/662a8d3b358f/fig_007_01.png" | relative_url }})

Decomposing the coding scores reveals where the failure lives: functional correctness stays high (>86% test-pass rate across all frontier models), but style compliance — the part requiring privilege-based conflict resolution — is the bottleneck. Models can write working code. They can't consistently figure out which style instruction wins.

## Fragile to Surface Form

Then there's the sensitivity finding, which is rougher than the headline numbers suggest.

Switching from ordinal to scalar PPI — same ordering, different format — drops GPT 5.4 by 8.4 percentage points and Opus 4.6 by 8.0. The models aren't reasoning about abstract priority. They're pattern-matching on how the priority looks in the prompt.

![Switching from ordinal to scalar Privilege Prompt Interface causes accuracy drops of 8+ percentage points in GPT 5.4 and Opus 4.6, despite identical relative ordering]({{ "/assets/figures/662a8d3b358f/fig_007_02.png" | relative_url }})

Even more telling: small numerical perturbations that preserve strict ordering — shifting each scalar value by a random $\delta \in [-3, +3]$ — cause per-sample flip rates of 8–17% across frontier models. The *relative* order hasn't changed. The *specific numbers* have shifted slightly. And models flip their answers.

Current models have a fragile, format-dependent grip on privilege reasoning. They don't reliably abstract "higher priority wins" away from surface cues.

## More Thinking Doesn't Save It

GPT 5.4 improves from 15.5% (no reasoning) to 60.9% (high reasoning effort) on the coding subset. That's a big jump, and it confirms that test-time compute helps.

But 60.9% on a task where human validation confirms at least ~80% is achievable is still a long way from solved.

And Claude models show an odd dip: at low reasoning effort, accuracy drops *below* the no-reasoning baseline. When thinking is fully disabled, Claude compensates by "thinking out loud" in its visible output. Low reasoning effort suppresses that visible chain-of-thought without replacing it with enough internal reasoning. More tokens doesn't fix what the model can't reliably track.

## The Safety Gap

ManyIH exposes a capability gap that standard IH evaluations hide. The >99% two-tier scores create a false sense of security. When privilege tiers scale to realistic numbers, models fail.

For deployed agentic systems — where instructions flow from system prompts, skill files, memory, tool outputs, other agents, and API responses — this gap is a safety problem. Prompt injection and jailbreak attacks exploit instruction conflicts. If models can't reason over even straightforward, non-adversarial priority conflicts, adversarial attacks get a free ride.

The PPI abstraction itself is immediately usable: privilege values are prompt-level, set at inference, no retraining required. That's a practical starting point. But the benchmark makes clear that *telling* models about priority and *having them follow* it are different problems.

Honest limits: ManyIH assumes privilege is assigned by a trusted party and uses non-adversarial prompts. It doesn't address adversarial privilege manipulation. The coding subset uses hand-curated style constraints; real-world conflicts can be subtler. And the benchmark isolates ManyIH from standard role-based IH — in practice, both hierarchies would operate simultaneously.

The ladder needs more rungs. Right now the models can't climb them.
