*TL;DR: You can give multi-agent LLM systems provable deadlock-freedom by writing a global coordination spec as message sequence charts and projecting it into local agent programs—and the same guarantee holds even when an LLM generates the workflow at runtime.*

## The Deadlock You Didn't See Coming

Picture a team of LLM agents collaborating on a medical diagnosis. One agent finishes early, sends its result to the orchestrator, and moves on. Another agent is still working.

The orchestrator receives the early result and tries to process it—before the second agent's verdict arrives. The system is stuck waiting for a message that hasn't been sent yet.

This isn't a theoretical worry. It's the same kind of deadlock that has plagued distributed systems for decades, now compounded by the nondeterminism of LLM outputs.

The frameworks we reach for today—LangGraph, AutoGen, CrewAI—give us infrastructure for building multi-agent systems, but no formal guarantees that coordination errors won't arise. Testing can't rule them out: the branching and interleaving space is too large, and LLM outputs are stochastic.

Bollig, Függer, and Nowak ask a different question: what if we could *prevent* coordination errors by construction, the same way communication protocols do?

## From MSCs to a Coordination DSL

Their starting point is the message sequence chart (MSC)—an ITU-standard notation for communication protocols that makes causal dependencies between agents visually explicit.

Each agent occupies a vertical lifeline; messages flow as arrows between lifelines; local actions appear as dots. An MSC captures one execution trace. A *global workflow* composes MSCs into a full protocol, specifying who talks to whom, in what order, and who owns each decision.

![MSC diagram showing the reviewed-execution protocol with four lifelines, Planner-owned conditional branch, action blocks for LLM calls, and message arrows coordinating the flow]({{ "/assets/figures/4160a217eb94/fig_005_01.png" | relative_url }})

The authors define a DSL that extends standard MSCs with two constructs designed for LLM agents:

- **Action blocks**: opaque, typed functions like `act LLM1 : (verdict, reason) = assess(notes, diagnosis)`. These represent LLM calls whose outputs are unpredictable. The language treats them as black boxes—no guarantees about what they return, only about the coordination around them.

- **Owned control flow**: every conditional and loop has an explicit decider. In `if plan_needs_review@Planner then ... else ...`, only Planner chooses the branch. In `while (not agreed and trials < max_rounds)@LLM1 do ... exit ...`, only LLM1 decides whether to iterate or exit.

Owned control flow is the key design choice. Every branching decision has a single authoritative source. No agent is left guessing which branch the system took.

## Projecting Down to Local Programs

Given a global workflow written in this DSL, the authors define a *syntax-directed projection* that produces one local program per lifeline.

The projection walks the global program structurally:

- Message statements become sends and receives on the relevant endpoints.
- Action blocks become local actions on the owning lifeline, and no-ops on everyone else.
- For owned control constructs, the decider keeps its source-level `if`/`while`, broadcasts its decision to every other participating lifeline, and each recipient branches based on a received control tag.

That broadcast is the subtle part. Consider the reviewed-execution example: Planner decides whether review is needed, and Orchestrator must learn the outcome before it knows whether to expect a critique. Without an explicit control message, Orchestrator might dequeue Executor's result before Reviewer's critique, producing a stuck state.

The projection inserts a tagged control broadcast—`send Planner(⊤, κ^P_ctrl) → Orchestrator` or `send Planner(⊥, κ^P_ctrl) → Orchestrator`—ensuring every participant learns the branch before proceeding.

The result: projection is always defined for well-typed workflows. No realizability check. No automaton construction. No branch-mergeability conditions (the consistency requirements that multiparty session types need).

> Owned control flow together with explicit control broadcasts eliminates the need for such consistency conditions.

global spec → local programs + control broadcasts → (erase broadcasts) → global spec

## How the Zipper Lemma Locks It In

The hardest technical challenge is *asynchronous sequential composition*. In a distributed run of `$P_1 ; P_2$`, different lifelines cross the `$P_1$`/`$P_2$` boundary at different times.

Some agents have finished `$P_1$` and started `$P_2$`, while others are still inside `$P_1$`. Can `$P_1$` always be completed consistently with whatever `$P_2$` work has already begun?

The **Zipper lemma** (Lemma 5) answers yes. Given a partial execution prefix, it constructs a completion of `$P_1$` that is compatible with the `$P_2$` suffix that has already started.

The proof is structural induction with a carefully designed postcondition that tracks each lifeline's position across the boundary. Control broadcasts make branch decisions locally recoverable—each lifeline can determine which choices were made, even asynchronously.

Deadlock-freedom (Corollary 1) follows directly. Given any prefix of a projected execution, apply the Zipper lemma with an empty suffix. Every partial run can be extended to a complete one.

## Letting LLMs Write Safe Workflows

The most practically compelling piece is the **runtime planning extension**. Sometimes you don't know the coordination structure in advance.

ZipperGen lets you declare a `@planner` action inside a workflow:

```python
@planner(
    description="A workflow planner for professional writing tasks.",
    lifelines=["Worker1", "Worker2"],
    allow=["llm", "if"],
    instructions="Use Worker1 to draft, Worker2 to assess quality; "
                 "route back to Worker1 for revision if needed.",
)
def write_document(request: str, job_desc: str, cv_sketch: str) -> str: ...
```

At runtime, a designated LLM generates a complete sub-workflow in the DSL. ZipperGen validates it for syntactic well-typedness, projects it, and executes it.

The generated sub-workflow runs in an isolated scope with its own threads and channels. It inherits the same structural guarantees:

> The planner needs only to produce a syntactically valid, well-typed global workflow. Deadlock-freedom is then guaranteed by the projection.

Asking an LLM to directly produce deadlock-free distributed code would be unreasonable. Asking it to produce a well-typed workflow in a constrained DSL—where deadlock-freedom comes for free—is far more tractable.

## What Actually Gets Proved—and What Doesn't

The guarantees are real but narrow. The framework proves coordination-layer properties: deadlock-freedom, message-matching, correct control-flow synchronization.

It does *not* guarantee that individual LLM actions produce correct outputs, that they terminate, or that the overall result is useful. The consensus example in the paper makes this explicit: the loop bound on `max_rounds` constrains the coordination pattern, not the termination of individual LLM calls.

The formal development covers only finite, terminating executions. Loops are assumed to eventually exit.

Extending to reactive or infinite-behavior settings would require new semantic machinery, which the authors flag as future work. That said, the by-construction approach sidesteps a whole class of realizability decidability problems that have plagued MSC and session-type frameworks for decades.

Instead of checking whether a global spec *admits* a deadlock-free implementation, you write it in a language where projection *always* produces one.

## Running It: ZipperGen

ZipperGen (open-source, at [zippergen.io](https://zippergen.io)) implements the full pipeline: write a `@workflow` in Python, and the decorator rewrites it into an internal representation mirroring the formal grammar, projects it, and executes it with one thread per lifeline connected by FIFO queues.

Control broadcasts use per-construct tags, keeping them disjoint from user messages and from each other across nested constructs.

A companion UI called ZipperChat renders executions as live MSC diagrams—essentially the same visual language as the formal semantics.

The medical diagnosis consensus example from the paper demonstrates the loop construct: two LLMs exchange verdicts and reasoning, iterating until they agree or hit a round limit, with deadlock-freedom by construction.

The deadlock never comes. By construction.
