*TL;DR: When multiple agents share one orchestrator context window, steering accuracy collapses to 21%. DACS fixes this with a simple mode switch: keep only compact status summaries by default, then inject exactly one agent's full context when it asks for help — achieving 90–98% accuracy across eight test scenarios.*

## Ten Agents Walk Into a Context Window

Ten agents running in parallel — one writing a BST implementation, one surveying transformer attention literature, one debugging CSV encoding. Each hits a decision point and asks the orchestrator for guidance.

The orchestrator has one context window. It holds all ten agents' task states, partial outputs, and pending questions at once. When agent a₁ asks "recursive or iterative traversal?", the context is swimming in vocabulary about attention heads and character encodings that has nothing to do with the answer.

The result: 21% steering accuracy.

Not because the model can't reason — because the wrong agents' context is competing for attention at the exact moment a decision gets made.

## The Pollution Problem

Multi-agent LLM orchestration is everywhere now — Claude Code Agent Teams, OpenCode, AgentOrchestra. They all share a common architecture: one LLM instance coordinates multiple agents through a single context window.

That window becomes a contested resource.

Prior work has addressed context bloat for a single agent. Cruz's Adaptive Focus Memory assigns fidelity tiers to messages within one agent's history. CodeDelegator separates planner and executor contexts at task-decomposition time. AgentOrchestra uses hierarchical routing to reduce how much context reaches the planner, and its authors explicitly note that flat coordination "tends to accumulate irrelevant context."

But none of these solve the runtime problem: when N concurrent agents are actively vying for steering, what does the orchestrator hold *at the moment of each decision*?

## The Idea: One Agent at a Time

DACS — Dynamic Attentional Context Scoping — introduces a simple state machine for the orchestrator with two core modes.

**REGISTRY mode.** The orchestrator holds only compact status summaries for every agent — one registry entry per agent, each capped at ≤200 tokens. It monitors all agents, responds to the user, and queues incoming steering requests. No full task context. No decision history. Just: "a₂: RUNNING, transformer survey, 3/5 steps done."

**FOCUS(aᵢ) mode.** When agent aᵢ emits a `STEERINGREQUEST`, the orchestrator transitions. It now holds F(aᵢ) — the full context of the requesting agent (task description, prior steering exchanges, current output) — plus compressed registry summaries for all other agents. The context window contains exactly what is needed to steer aᵢ, and nothing from any other agent's task thread.

Formally, the focus context is:

$$C_{\text{focus}} = F(a_i) + R_{-i}$$

where $R_{-i}$ is the registry minus aᵢ's entry (since aᵢ's full context is already in $F(a_i)$). No scoring function. No probabilistic compression. The construction is deterministic.

This is the core move: *agent-triggered, asymmetric, exact isolation*.

## How It Stays Focused

The orchestrator never holds two agents in FOCUS simultaneously.

It's a finite-state machine with three states: REGISTRY, FOCUS, and USERINTERACT. The transitions are explicit. When an agent needs help, it emits a `STEERINGREQUEST` with an urgency level.

MEDIUM requests queue up. HIGH urgency can *interrupt* an active focus session — the current state is saved, the new agent gets priority, and the interrupted session resumes after. User messages never get dropped; they queue during FOCUS and process on the next REGISTRY entry.

The context builder enforces a hard token budget before every LLM call. If $F(a_i)$ plus the registry exceeds budget, lower-urgency registry entries get truncated first. The focused agent's context is never truncated.

### The Scaling Math

The focus context grows sub-linearly in N. Only the registry adds tokens per additional agent:

$$|C_{\text{focus}}| \approx |F| + (N-1) \cdot |r|$$

where $|F| \approx 450$–$550$ tokens and $|r| \approx 25$ tokens.

Empirically: 561 tokens at N=3, 633 at N=5, 816 at N=10. The efficiency ratio against a flat context grows from 2.12× to 3.53× as N increases. More agents means more context pollution in the baseline — but DACS barely notices.

## The Numbers

Across eight synthetic scenarios and 160 trials, DACS steering accuracy ranges 90.0–98.4%. The flat-context baseline? 21.0–60.0%. Every comparison is significant at p < 0.0001.

The pattern is consistent and the gaps are dramatic:

- **N=3:** DACS 96.7%, baseline 60.0% (+36.7 pp)
- **N=5:** DACS 96.7%, baseline 38.7% (+58.0 pp)
- **N=10:** DACS 90.0%, baseline 21.0% (+69.0 pp)

Wrong-agent contamination drops from 28–57% down to 0–14%. The advantage *grows* with more agents, not shrinks.

### Diversity Doesn't Save the Baseline

Phase 2 tests three heterogeneity structures. Even in a *cascade* scenario — five pipeline-dependent agents where you'd think seeing all outputs would help — DACS wins by +37.3 pp. The flat baseline does not benefit from cross-agent visibility. It just drowns in it.

In the *crossfire* scenario with maximally diverse, vocabulary-disjoint domains, DACS hits 96.0% with 0.0% contamination. The baseline manages 37.0% with 53.0% contamination.

### Decision Density Amplifies the Gap

Here's the most striking finding. Quintuple the decisions per agent from D=3 to D=15 and the baseline accuracy *falls* from 60.0% to 44.2%. DACS accuracy *rises* from 96.7% to 98.4%.

More history in the focus context is additive signal. More history in a flat context is noise.

![Steering accuracy vs decision density at N=3, showing DACS improving with more history while the baseline degrades]({{ "/assets/figures/90e1d257a973/fig_010_03.png" | relative_url }})

### Real Agents Confirm It

Phase 4 replaces all scripted stubs with autonomous LLM agents (Claude Haiku 4.5) generating free-form questions — no templates, no keyword games. Two independent LLM judges (Haiku and GPT-4o-mini) agree: DACS leads by +17.2 pp at N=3 and +20.4 pp at N=5.

The advantage grows with N, mirroring the synthetic results exactly.

![DACS vs baseline steering accuracy with real LLM agents at N=3 and N=5, confirmed by two independent judges]({{ "/assets/figures/90e1d257a973/fig_012_04.png" | relative_url }})

## The Honest Limits

Three things to keep straight.

Phase 4 only covers N∈{3, 5} at low decision density with one model family. We don't yet know how DACS behaves at N=10 with real agents, or with decision density cranked up to D=15.

The contamination metric (M2) conflates helpful registry-grounding mentions with genuine cross-agent leakage in the real-agent setting. When the orchestrator says "unlike a₂'s task, your BST walk benefits from…", the binary detector flags it. That's a measurement problem, not a fidelity failure — DACS's decision quality (M1) stays significantly above baseline despite the elevated M2 reading.

The interrupt protocol for high-urgency preemptions was exercised naturally but never ablated in isolation. We know it works; we don't know how much it contributes.

## What This Means

DACS is a *deterministic* mechanism. No scoring functions to tune, no retrieval indices to maintain, no architectural restructuring required. Any multi-agent orchestrator that exposes a context builder can adopt it — swap in the state machine, wire up the focus mode, done.

It's also complementary to hierarchical routing. AgentOrchestra reduces how many agents reach the orchestrator. DACS controls what the orchestrator holds *at the steering moment*. A production system could use both.

The deeper lesson: what's in the context window at the moment of decision is the dominant variable for multi-agent steering accuracy. Not model size. Not topology. Not compression. Just isolation.

One agent. One focus. One clean shot at the right answer.
