*TL;DR: An 8B-parameter multi-agent system with a single orchestrator and tiny sub-agents matches or beats a 32B single-agent model on tool-use benchmarks — and the secret isn't bigger models, it's putting the reasoning at the top of the hierarchy.*

## The Setup

An 8B model walks into a benchmark. It has three specialist sub-agents — a Web Searcher, a Coder, and a File Inspector — and one orchestrator that tells them what to do.

Across the hall, a 32B model works alone with direct tool access. Same benchmarks. Same tools.

The 8B team ties on GAIA (23.0 vs. 23.0) and wins on AIME (55.0 vs. 45.0). This isn't a fluke. It's architecture beating scale.

Żywot et al. ask a disarmingly simple question: can small agents, organized well, outperform a much larger model working alone?

Their answer is yes — and the reason is not what you'd expect.

## Why This Experiment Needed Running

Scaling model size is expensive and does not reliably improve multi-step reasoning or tool use (Shojaee et al., 2025; Li, 2026). Multi-agent systems promise a cheaper alternative, but prior frameworks like MetaGPT deploy large models throughout the hierarchy and never isolate which design choices actually drive performance.

Without controlled comparisons, you can't tell whether the win comes from collaboration itself or from just throwing more parameters at the problem.

This paper runs the experiment that needed running: same model family, same tools, same benchmarks, systematically varying who thinks and how big each component is.

## How the System Works

The system is deliberately minimal.

One orchestrator plans, decomposes tasks, and delegates to three specialist sub-agents. No sub-agent-to-sub-agent communication.

A shared structured memory stores key intermediate results — query analysis, previous steps, tool outputs, sub-goals — so the orchestrator can reconstruct state without replaying full conversation histories.

This star topology, where everything flows through the orchestrator, cleanly separates planning from execution. It also makes ablation straightforward.

The benchmarks span five tool-intensive domains: GAIA (end-to-end tool use), GPQA (graduate-level science), AIME (mathematical problem-solving), MuSiQue (multi-hop retrieval), and HLE (expert-level knowledge). All experiments use the Qwen3 family at 1.7B, 8B, and 32B.

## Where You Put the Reasoning

The central variable is *where* explicit thinking happens. The authors test four configurations: no thinking, sub-agent-only thinking, orchestrator-only thinking, and thinking everywhere.

The result is striking. Orchestrator-only thinking yields the largest and most consistent gains across every benchmark: +18.2 on GAIA, +36.7 on AIME, +10.0 on MuSiQue over the no-thinking MAS baseline.

![Accuracy across five benchmarks for different system configurations, comparing single-agent direct tool use against multi-agent systems with varying thinking placements]({{ "/assets/figures/139b58dfca90/fig_003_01.png" | relative_url }})

Sub-agent-only thinking? It slightly *increases* tool calls on GAIA (447 → 483) while degrading performance.

The asymmetry makes sense once you see it. The orchestrator governs global planning and task decomposition, so reasoning there directly improves every downstream decision.

Sub-agents execute localized subtasks. Adding a reasoning trace to each one introduces verbosity and coordination overhead without improving execution quality.

$$\text{Orchestrator thinking: turns } 3.6 \rightarrow 2.0,\ \text{tool calls } 2.7 \rightarrow 1.1$$

The orchestrator plans before it acts, and that plan halves the work. Sub-agent thinking adds +6.1 seconds of latency per query; orchestrator thinking adds only +0.6 seconds. Better planning more than pays for its cost.

## Planner-Limited, Not Executor-Limited

The capacity-swap experiments are the paper's sharpest result. The authors pair 8B and 32B orchestrators with sub-agents ranging from 1.7B to 32B.

With orchestrator thinking enabled, the 8B orchestrator scores 23.0, 23.0, and 23.6 on GAIA with 1.7B, 8B, and 32B sub-agents respectively. Nearly flat across a 19× range in sub-agent size.

![GAIA accuracy for different sub-agent sizes paired with an 8B orchestrator, showing nearly identical performance regardless of sub-agent capacity when orchestrator thinking is enabled]({{ "/assets/figures/139b58dfca90/fig_006_02.png" | relative_url }})

Without thinking, sub-agent size matters more — but once the planner can reason, you can scale executors down to a fifth of the orchestrator's size and lose almost nothing.

This flips the standard scaling intuition. Multi-agent systems are *planner-limited*, not executor-limited.

Pour compute into the planner, not the workers.

## The Efficiency Edge

The MAS with structured memory cuts prompt tokens by 43% compared to the direct-tool configuration. Instead of re-feeding full conversation history every turn, the orchestrator builds a compact two-message prompt: system instruction plus a reconstructed state summary.

![Prompt token usage comparison showing the multi-agent system with structured memory uses 43% fewer tokens than the single-agent direct configuration]({{ "/assets/figures/139b58dfca90/fig_006_03.png" | relative_url }})

The 8B MAS without thinking runs 4.2× faster than the 32B direct configuration despite making more model calls. Compact per-turn prompts, short single-turn sub-agent dispatches, and faster inference on the smaller model more than offset the overhead of coordination.

## Honest Limits

Three caveats matter.

First, all experiments use Qwen3 models. Whether these findings generalize to other architectures and training regimes is genuinely unknown.

Second, the star topology deliberately excludes richer coordination schemes. Sub-agents never talk to each other, which may underrepresent what multi-agent collaboration can achieve.

Third, HLE remains unsolved across all configurations, suggesting that some tasks exceed current systems regardless of how you organize them.

The paper also documents clear failure modes. Tools hurt when retrieval overrides correct parametric knowledge. Orchestrator thinking hurts when reasoning substitutes internal rationalization for external verification. Over-deliberation is a real thing.

## Scale the Planner

You don't need bigger workers. You need a better foreman.
