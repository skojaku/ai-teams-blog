*TL;DR: A 7B-parameter LLM trained with reinforcement learning on just nine ML tasks rivals GPT-5 at autonomous machine learning engineering — at less than one-twentieth the cost per trajectory.*

## Nine Tasks, Twenty Times Cheaper

A 7-billion-parameter model just went head-to-head with GPT-5 on autonomous ML engineering.

Not a distilled knockoff, not a prompt-engineered trick. A Qwen2.5-7B model that *learned* from its own experience, using reinforcement learning, across only nine training tasks.

It beat DeepSeek-R1 (671B) and landed within striking distance of GPT-5 — for under a penny per trajectory.

The paper calls this the shift from *prompt-based agentic ML* to *learning-based agentic ML*. The name is descriptive but the insight is sharp: if you want a small model to do autonomous ML well, stop prompting it and start training it.

![Framework overview showing exploration-enriched SFT building a diverse action pool, step-wise RL training on single actions sampled from expert trajectories, and the reward module unifying ML feedback signals]({{ "/assets/figures/0fa794cd4edb/fig_004_01.png" | relative_url }})

## The Old Way Has Two Problems

Current agentic ML systems like AIDE and ML-Master work by giving a large LLM a prompt scaffold and letting it iterate. Edit code, run experiment, read feedback, repeat.

This works — if you can afford it. GPT-5 with the AIDE scaffold hits ~21% average performance gain on held-out tasks. But each trajectory costs more than 20× what ML-Agent's does, and you're locked into a proprietary API.

Go smaller and the problem flips. A Qwen2.5-7B-Instruct under the same AIDE scaffold barely reaches 1% average gain. The model can't internalize what worked and what didn't across runs.

The constraint is fundamental: prompt-based agents don't learn. They execute instructions. Every trajectory starts from the same static prompt, armed with the same static strategies, oblivious to what the last hundred runs taught.

## Making a Small Model Explore

The first thing ML-Agent needs is the ability to *try different things*.

Left to its own devices, a 7B model exploring an ML task will repeatedly make small, conservative edits — tweak a learning rate, add a dropout layer, swap one optimizer for another. Across episodes, the trajectories look like photocopies.

The fix is exploration-enriched fine-tuning. The authors organize ML optimization ideas into three buckets — data, model, and learning — and have an LLM generate a large pool of candidate strategies within each.

An embedding-based diversity filter prunes redundant ideas. Then GPT-4o-mini executes combinations of these ideas on fast ML tasks, producing 10k trajectories.

The 7B model is then supervised-fine-tuned on these diverse trajectories:

$$\mathcal{L}_{\text{SFT}}(\theta) = -\mathbb{E}_{\tau \sim \mathcal{D}} \left[ \sum_{t=0}^{n-1} \log \pi_\theta(a_t | s_t) \right]$$

This doesn't teach the model *which* actions are good. It teaches the model *what kinds of actions exist* — data augmentation, architecture changes, loss function swaps, training schedule tweaks — so that when RL begins, it has a wide menu to sample from.

Remove this step and performance collapses: average gain on held-in tasks drops from 18.01% to −0.66%, on held-out tasks from 15.91% to −6.20%. The model hasn't learned to explore, so RL has nothing to work with.

## Don't Roll Out the Whole Experiment

Standard PPO needs full trajectory rollouts. In ML engineering, a single trajectory means running a full pipeline — data loading, training, evaluation — which can take minutes to hours per step. You'd need weeks of GPU time to collect enough rollouts for meaningful updates.

ML-Agent sidesteps this entirely. Instead of rolling out complete trajectories, it samples single states from a pre-collected pool of expert trajectories, then evaluates *one action* at that state.

The objective becomes:

$$J_{\text{step}}(\theta) = \mathbb{E}_{s_t \sim d^{\pi_e}, \; a_t \sim \pi_\theta(\cdot|s_t)} \left[ R(s_t, a_t) \right]$$

The state distribution $d^{\pi_e}$ comes from the expert, not from the current policy. This decouples state sampling from the training loop entirely. You pre-collect 10k states once, then train PPO on single-step decisions.

Two things happen: training runs in hours instead of weeks, and the agent learns from targeted single-step credit assignment rather than sparse trajectory-level feedback.

The ablation is stark — episode-wise RL (standard PPO) achieves only 2.86% average gain on held-out tasks. Step-wise RL reaches 15.91%.

## The Reward Has to Handle Everything

ML experiments produce messy feedback — clean validation metrics, compilation errors, out-of-memory crashes, read-only queries that change nothing.

The reward module maps all of this to a single scalar:

$$R(s_t, a_t) = \begin{cases} -1 & \text{if } a_t \notin \mathcal{A}_{\text{valid}} \text{ or } f_t \in \mathcal{F}_{\text{error}} \\ 0 & \text{if } a_t \in \mathcal{A}_{\text{valid}} \setminus \mathcal{A}_{\text{edit}} \text{ or } f_t \in \mathcal{F}_{\text{corner}} \\ \frac{m_{t+1} - m_t}{m_{\text{best}} - m_{\text{init}}} & \text{if } a_t \in \mathcal{A}_{\text{edit}} \text{ and } f_t \in \mathcal{F}_{\text{success}} \end{cases}$$

Invalid actions and errors get −1. Valid but non-editing actions (like reading a file) and corner cases (out-of-memory) get 0. Successful code edits get a normalized performance improvement, scaled by the gap between the initial baseline and the best human-achievable score.

The scaling matters. When only the normalized performance signal is kept — no format penalty, no corner-case handling — cifar10 drops from 33.80% to 17.58%.

The format penalty matters even more. The paper reports that removing format constraints causes the largest single degradation, including a −11.75% hit on cifar10.

Corner-case handling keeps training stable when the environment misbehaves. The paper notes it has minimal impact in practice due to rarity, but prevents over-penalizing non-fatal issues.

## The Numbers

On 10 held-out tasks (never seen during training), ML-Agent achieves 16.40% average performance gain.

DeepSeek-R1 with AIDE gets 11.40%, with ML-Master 13.13%, with MLAB 6.83% — all using a 671B model.

GPT-5 with AIDE reaches 20.95% — the strongest proprietary baseline. Our 7B agent comes within ~4.5 points.

Cost per trajectory: under $0.01 for ML-Agent. That's more than a 20× gap against proprietary-model baselines for comparable performance.

![Cost versus performance comparison showing ML-Agent in the top-left corner: competitive gain at a fraction of the cost of GPT-5 and DeepSeek-R1 agents]({{ "/assets/figures/0fa794cd4edb/fig_018_06.png" | relative_url }})

On held-in tasks, ML-Agent hits 18.01% average gain. Scaling RL training from 0 to 3 to 6 to 9 tasks monotonically improves both held-in and held-out performance, with held-out gains climbing from ~0% to ~16%.

## The Ceiling and the Floor

The expert trajectories used for both SFT and step-wise state sampling came from GPT-4o-mini. That means the 7B model's exploration ceiling is bounded by what GPT-4o-mini could produce.

A stronger expert would likely yield a stronger agent. But the RL phase pushes beyond mere imitation — the paper's training curves show continuous improvement from SFT baseline through RL on both held-in and held-out tasks.

The generalization story spans 10 held-out tasks chosen by the authors from MLE-bench — diverse in modality and metric, but not a full benchmark sweep. How the agent handles truly novel task types, like reinforcement learning or generative modeling, remains untested.

And there is setup cost. You need to collect expert trajectories, run SFT, then run RL on 8 A100 GPUs before your agent is ready. This is a training-time investment, not a zero-shot prompt.

But it pays for itself. Once trained, the model runs at a fraction of the per-query cost of any proprietary alternative.

## What This Opens

The core claim is architectural, not just empirical: *agentic ML can be a learning problem, not a prompting problem.*

Prior work showed that RL-trained LLM agents can improve at sequential decision-making tasks. ML-Agent shows the same principle works for an even richer feedback environment — one where code execution, metrics, and error messages provide dense, informative reward signals that prompt-only agents simply ignore.

The step-wise RL trick is general. Any long-horizon agentic domain where full rollouts are expensive — software engineering, scientific computing, data analysis — could benefit from decoupling state sampling from policy updates.

And the cost curve bends in the right direction. More training tasks improve generalization monotonically. A 20× cost reduction over proprietary models at comparable quality means you can run orders of magnitude more experiments, more iterations, more exploration — the very thing that makes ML engineering work in the first place.

A 7B model that learned to do ML by doing ML.
