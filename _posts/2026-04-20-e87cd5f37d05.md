*TL;DR: Frontier LLM agents satisfy hard constraints 70–90% of the time—but land in the top tier of feasible solutions only 20–50%. COMPASS reveals that the bottleneck isn't tool use; it's that agents don't explore the search space strategically enough.*

## A Gap You Can Drive Through

Ask GPT-5 to book you a trip that stays under budget, and it will probably do it.

Ask it to find you the *cheapest* trip under budget? That's where things fall apart.

COMPASS, a new benchmark from Qin, Bai, and colleagues at Apple and Harvard, exposes a consistent pattern across every frontier model: high feasibility, low optimality. Closed-source agents produce feasible solutions on 70–90% of tasks—respectable. But when you check whether those solutions rank among the best feasible options, the rate drops to 20–50%. The gap is real, and it is wide.

The shift this paper forces is a reframing: *satisfying constraints ≠ solving the user's problem.*

## Why Existing Benchmarks Miss the Point

Existing benchmarks test pieces of what an agent should do, but not the whole job.

TravelPlanner (Xie et al., 2024) asks whether an agent can produce a plan that meets hard constraints—budget, dates, room type—but never checks whether that plan is *good* relative to all feasible alternatives. τ-bench (Yao et al., 2024) evaluates tool use and multi-turn dialogue, yet its tasks require only short-horizon reasoning, not strategic search over a combinatorial space. PlanBench (Valmeekam et al., 2022) tests planning through state transitions, but without tool-mediated exploration or user interaction.

None of these ask the question that matters most when you deploy an assistant: once the agent knows what you need, can it find you the *best* option?

COMPASS closes that gap by framing agent evaluation as constrained optimization:

$$\min_{x \in X} f(x) \quad \text{s.t.} \quad g_i(x) = c_i, \quad h_j(x) \geq d_j$$

Here $x$ is a travel itinerary (a tuple of specific hotel, flight, and permit bookings), the constraints define the feasible set $X_{\text{feas}}$, and $f(x)$ is the user's utility function—the thing to optimize.

Feasibility asks: is $x \in X_{\text{feas}}$? Optimality asks: where does $f(x)$ rank among all feasible options?

## How COMPASS Works

### 281 tasks, real data, ground-truth optima

The benchmark builds 281 tasks across three difficulty levels. Level I tasks ask the agent to optimize a single service—pick the cheapest hotel that meets the user's constraints. Level II adds flights: now the agent must coordinate dates and prices across two services. Level III chains in permits—scarce, date-locked bookings that cascade constraints through the whole itinerary.

Two types of utility objectives keep evaluation unambiguous. Single-metric optimization asks the agent to minimize one continuous quantity (total price). Feature-count maximization hands the user a wish list—"pet-friendly, free breakfast, direct flight"—and scores by how many checkboxes the solution hits.

Ground-truth optima come from exhaustive enumeration: every task's full feasible set is computed via SQL filters over the database, then ranked by $f(x)$.

No LLM-as-judge. No "looks reasonable" scoring. The answer is either objectively optimal or it isn't.

### Real databases, 18 tools, 108 user personas

COMPASS doesn't use synthetic toy data. The environment pulls 100K+ hotel offers, 67K+ flight offers, and 50 permit types from Booking.com via RapidAPI. Hotels aren't single entries—they're bundles of room types with distinct cancellation policies and amenities. Flights vary by carrier, class, layovers, and departure times.

Agents interact with 18 tools across four categories: hotel search and details, flight search and details, permit availability, and utility helpers (calculator, calendar, notebook). Coding agents can also write and execute Python scripts that call these tools programmatically.

The user simulator—backed by GPT-5—carries multi-turn conversations across 108 distinct personas. Four orthogonal behavioral axes define each persona: how quickly they reveal constraints (upfront vs. progressive), how they communicate (concise, emotional, redundant), how carefully they check the agent's work, and how much they demand explicit reasoning before accepting a recommendation. This makes constraint discovery a first-class challenge, not a setup step.

## The Feasible-Optimal Gap

The headline finding is stark. Across every closed-source model tested—including GPT-5, Claude Opus 4.6, Claude Sonnet 4.6, and Gemini 2.5 Pro—feasibility rates land between 70% and 90%. Optimality rates at the top tiers of all feasible solutions land between 20% and 50%.

The gap persists even though users explicitly request optimization. The task prompt says "find the cheapest option" or "maximize amenities." The agent understands the ask. It just doesn't search hard enough for the answer.

![Bar chart showing the feasible-optimal gap across frontier models: feasibility rates cluster at 70–90% while optimality rates drop to 20–50%, with coding agents partially closing the gap]({{ "/assets/figures/e87cd5f37d05/fig_006_02.png" | relative_url }})

### Open-source falls off a cliff

The Qwen3 family (8B, 14B, 32B) shows competitive performance on Level I hotel-only tasks. But at Levels II and III, feasibility plummets. The failure mode shifts: open-source models can't even find a valid plan once multiple services need coordination, let alone optimize it.

## The Bottleneck Isn't Tool Use

Tool-call error rates for frontier models sit below 1%.

Claude Opus 4.6 (Code) and Claude Sonnet 4.6 (Code) both hit 0.10% and 0.40% total error rates respectively. These agents know how to call the tools—they pick the right function, pass valid parameters, and get back clean results.

The bottleneck is *exploration*.

The authors quantify this with two metrics: search effort $E$ (number of unique, correct tool-call signatures) and search yield $Y$ (number of unique bookable offers retrieved). Stronger models consistently show higher $E$ and higher $Y$. Models that discover more options are more likely to find optimal ones.

But quantity alone doesn't guarantee quality. When Claude Sonnet 4.6 (non-code) does manage high search effort, its optimality rate *drops*—it gathers information but can't synthesize it. The coding variant handles the same volume without degrading, suggesting that code-based systematic search is a viable path forward.

## Search Smarter, Not Harder

A case study makes the gap vivid. Given the same "cheapest trip to Yellowstone in August" prompt, GPT-5 and Claude Sonnet 4 behave very differently.

GPT-5 first checks which day of the week candidate dates fall on, then deliberately targets Tuesday–Friday departures because mid-week travel is typically cheaper. It samples flights across early, mid, and late August for broad coverage—strategic sampling, not brute force. It articulates its reasoning transparently, giving the user a foothold for feedback.

Claude Sonnet 4 searches exactly two arbitrary dates (August 5 and 12) with no justification. It misaligns hotel check-in dates with flight departures—a temporal coordination error that violates a hard constraint by itself. The reasoning trace is generic and uninformative.

The difference isn't effort. It's *strategy*. GPT-5 applies heuristics that are actually consistent with the data (mid-week trips are cheaper on average in this database). Claude Sonnet 4 throws darts at the wall.

## How Code Closes the Gap

Giving models code execution tools closes part of the gap. Claude Sonnet 4.6's OR-10% rises from 57% to 66% when it can write and run scripts that systematically call the tool APIs. Coding agents consistently outperform their non-coding counterparts across all model families.

But code doesn't make the problem trivial. The agent still has to discover which tools exist, understand what the data returned by each tool means, and write code that orchestrates them coherently. And COMPASS's tool outputs are clean and structured; real-world data from web interfaces would be far noisier.

## How Users Make It Harder

Two user-side factors degrade performance across all models.

Delayed constraint revelation—users who drip-feed requirements over multiple turns instead of stating them upfront—causes consistent performance drops. The more turns it takes the user to fully specify the task, the worse the agent does.

Communication style matters too. GPT-5 is the most robust model across nine different persona styles, with a coefficient of variation of just 12.85%. Smaller and open-source models swing wildly depending on whether the user is polite or curt.

One thing that *doesn't* help: asking the agent to double-check its answer. The trust axis—whether the user demands explicit reasoning before accepting a recommendation—produces no statistically significant improvement for any model. Re-checking without new information changes nothing.

## What COMPASS Can't Tell Us

COMPASS is travel-planning-specific with a single database snapshot. Whether these findings generalize to shopping, scheduling, or live web data remains an open question. The ground truth relies on exhaustive enumeration, which won't scale to larger combinatorial spaces. And the user simulator, while validated by human annotators (median clarity and appropriateness scores of 4 out of 5, with only 4% constraint-revelation error), covers a finite set of scripted behavioral axes.

The deeper limitation is also the deeper opportunity. COMPASS makes clear that the next frontier for LLM agents isn't better tool calling—it's strategic exploration. Agents need to maintain a running best option, refine their search around promising regions, and allocate their query budget adaptively. The benchmark gives the community a place to measure whether that's actually happening.

Good enough isn't optimal—and now we can tell the difference.
