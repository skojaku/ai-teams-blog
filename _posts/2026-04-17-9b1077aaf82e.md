*TL;DR: OccuBench uses LLMs as simulators of professional tool environments — emergency triage, nuclear safety monitoring, customs processing — making any domain evaluable without building real infrastructure, and reveals that every frontier model has its own occupational specialty profile.*

Can an agent triage patients in an ER? No environment exists. Manage a nuclear reactor safety alert? No benchmark covers it. Process customs declarations? No API is available.

These aren't edge cases. They're the default. Most high-value jobs for AI agents have zero benchmarks because no one can build the environments.

OccuBench flips the question. Not "how do we build the environment?" but "which LLM already understands the domain well enough to *be* the environment?"

## The Untestable Majority

Current agent benchmarks cover a vanishingly small slice of professional work.

WebArena evaluates web browsing. OSWorld evaluates desktop operations. SWE-bench evaluates code repositories. TAU-bench covers retail and airline APIs. Every single one is confined to domains with deployable, public infrastructure.

GDPVAL took a step toward real occupations — 44 of them, 1,320 tasks — but grades deliverable quality by rubric rather than testing interactive multi-step tool use.

Claw-Eval goes deeper with 300 human-verified tasks and trajectory-aware grading, but without cross-industry breadth.

The vast majority of professional domains remain invisible to evaluation.

And there is a second blind spot. None of these benchmarks test what happens when the environment breaks.

## Turning LLMs into Environments

OccuBench's core move is the Language Environment Simulator, or LES. Instead of deploying a real hospital information system, you configure an LLM with four things: a system prompt encoding simulation rules, a tool schema defining the action space, an initial state in JSON, and semantic annotations for causal consistency.

The LLM becomes a stateful, interactive environment:

$$
(s_{t+1}, o_{t+1}) = f_\theta(s_t, a_t; c)
$$

where $c$ is the configuration, $s_t$ is latent state maintained through the context window, $a_t$ is the agent's tool call, and $o_{t+1}$ is the structured JSON response. As long as an LLM understands the operational logic of a domain, it can simulate tool responses for that domain.

![Architecture of the Language Environment Simulator (LES). The agent issues tool calls; the LES generates observations conditioned on its configuration and conversation history. A rubric-based verifier scores the resulting trajectory.]({{ "/assets/figures/9b1077aaf82e/fig_005_01.png" | relative_url }})

Environment construction turns from an engineering problem into a configuration problem. "Domains with public environments" → "any domain an LLM can understand."

## How the Benchmark Is Built

A multi-agent synthesis pipeline powered by Gemini-3-Flash-Preview generates everything: environment configs, task instructions, tool definitions, solution plans, and verification rubrics.

Each task is executed multiple times — with and without a reference plan — to verify solvability and calibrate difficulty.

A majority-vote verifier scores trajectories against rubrics. A repair module diagnoses and fixes failures before re-execution.

After quality filtering — removing trivially easy, unsolvable, and invalid-schema instances — OccuBench contains 382 evaluation instances across 100 professional task scenarios, 10 industry categories, and 65 specialized domains. Each scenario maps to a real human job role: emergency triage nurse, operations engineer, customs officer, production scheduler.

Tasks average 5.5 tools and 16.2 tool calls.

## Two Ways We Test

OccuBench evaluates agents on two axes.

**Task completion** across 10 industry categories, revealing each model's occupational capability profile rather than a single aggregate score.

**Environmental robustness** under three fault injection regimes. All data is synthesized clean (E0). Faults are injected by appending rules to the LES system prompt at evaluation time:

- **E1 (Explicit):** HTTP 500, TimeoutError, ConnectionRefused — the agent knows the call failed. Retry.
- **E2 (Implicit):** Truncated data, missing fields, empty values — the response looks superficially correct. The agent must independently detect quality degradation.
- **E3 (Mixed):** Roughly half explicit, half implicit.

All faults are transient. Retrying recovers normal results. Two parameters control severity: fault count (default 2 events) and fault duration (default 2 consecutive tool calls per event).

## No Model Dominates Everything

GPT-5.2 leads overall at 79.6%.

But it scores only 67% in Commerce — 14 points below Qwen 3.5 Plus at 81%. Gemini 3.1 Pro tops Education (84%) and Science (81%) but struggles in Healthcare (62%). Claude Opus 4.6 leads Transportation (77%) while trailing Commerce (53%).

Open-source models are highly competitive. Qwen 3.5 Plus (69.9%) and DeepSeek V3.2 (69.6%) rank 4th and 5th, outperforming most Claude variants.

Each model has a distinct occupational profile. Gemini excels in knowledge-intensive domains. Claude Opus excels in operational domains that reward careful state tracking. Qwen excels in consumer-facing domains.

![Radar chart showing each frontier model's performance profile across 10 industry categories. Every model has a distinct shape — none dominates all domains.]({{ "/assets/figures/9b1077aaf82e/fig_009_03.png" | relative_url }})

Aggregate leaderboard rankings hide these differences. Pick your model for the job you need done.

## The Silent Fault Problem

Here is the counterintuitive result: implicit faults are harder than both explicit faults and mixed faults.

Average performance under E2 (implicit) is 53.4%, compared to E1 (explicit) at 62.6% and E0 (clean) at 67.5%. Four out of nine models score worse under pure implicit faults than under mixed faults. The explicit error component in E3 partially compensates — the agent at least knows something went wrong and can retry.

![Completion rates under clean (E0) and fault-injected (E1–E3) environments. Implicit faults cause the deepest performance drops across all models.]({{ "/assets/figures/9b1077aaf82e/fig_010_04.png" | relative_url }})

Under implicit faults, the agent gets no error signal. The response looks fine.

Most models simply trust it and move on. Claude Opus 4.6 drops 17.6 points from E0 to E2. Qwen 3.5 Plus drops 18.3 points.

If your production environment silently degrades data, your agent will likely not notice.

## Reasoning Effort Pays Off

GPT-5.2 improves by 27.5 points from the lowest reasoning effort (54.7%) to the highest (82.2%).

Larger models consistently outperform smaller variants within every family — gaps of 11.0% for Gemini, 10.2% for Qwen, 7.1% for Claude. Newer generations outperform older ones.

Allocating more compute to reasoning at inference time is a reliable lever for professional tasks.

## Strong Agents Make Weak Simulators

GPT-5.2 ranks first as an agent — and last as a simulator. When the LES is GPT-5.2, all agents average only 29.3%. Under Gemini Flash as simulator, they average 67.9%.

The paper documents three simulator failure modes with concrete examples.

*State fabrication:* GPT-5.2 invents two extra empty rooms in an emergency department, so the agent routes a patient to the wrong room.

*Entity omission:* GPT-5.2 drops a Database specialist from an escalation roster, making the correct handoff impossible.

*Rule invention:* GPT-5.2 independently enforces a 30-day return window expiration that is not in the task specification, rejecting a valid return.

In all three cases, the agent's strategy is correct. It fails only because the simulator violates the environment contract.

With a capable simulator, rankings stabilize. Qwen 3.5 Plus and Gemini Flash agree on 85.7% of pairwise comparisons, with the top-3 agents matching exactly.

The simulator is not a neutral observer — it is part of the evaluation apparatus. Verify its quality before trusting the rankings.

## Honest Limits

LESs model domain logic, not domain data. An LES knows that a drug interaction check should return contraindications, but the specific values are generated, not retrieved from a real database.

OccuBench evaluates decision-making — whether the agent checks the right things in the right order — not numerical precision. Financial calculations to the cent still need real-environment validation.

And the results depend on which simulator you use. Tasks verified as solvable under one LES may become unsolvable under another. Agent rankings shift. The simulator is an inherent variable in any LES-based evaluation, not a background detail you can ignore.

The untestable majority is now testable — as long as you remember who is doing the testing.
