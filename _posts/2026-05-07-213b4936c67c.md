*TL;DR: When LLM agents share durable memory, false claims can propagate like heritable traits—this paper argues that deciding which memories become shared institutional state is a selection problem, and proposes governance regimes that make that decision explicit and inspectable.*

## When Hallucination Becomes Heredity

Before the authors formalized a memory governance protocol, their multi-agent ecosystem had an auto-memory file sitting quietly in durable storage. It contained three false entries—plausible enough to survive until a later audit pulled them out.

The problem wasn't that an agent hallucinated. The problem was that the hallucination acquired *persistence*. It could be reloaded. It could shape another agent's decisions. It had become, in the authors' deliberate phrasing, heritable.

This is the shift that Cuadros, Maiga, Meskhidze, and Curtis-Trudel want us to see. Memory is turning LLM agents from stateless chatbots into state-bearing system components—and once a memory is durable enough to reload, influential enough to shape behavior, and portable enough to affect other agents or sessions, the core question changes.

It's no longer *how* agents remember. It's *which memories should be allowed to shape future behavior at all*.

## The Question Access Control Can't Answer

Park et al.'s generative agents showed that memory streams are a first-class resource for producing believable autonomous behavior. Rezazadeh et al. introduced shared memory tiers with dynamic access control for multi-agent LLM systems. Lam et al. tackled the risks and safety governance of evolving agent memory.

But access control asks who can read or write. Safety governance asks how to prevent contamination. Between them, a further question sits underdeveloped: *who or what decides that a candidate memory should become shared institutional state?*

The authors call this gap *governed collaborative memory*. Their argument is precise: memory governance is not merely access control, summarization, or retrieval policy. In persistent multi-agent systems, governance functions as a **selection regime**—filtering candidate memory variants so that only selected memories become durable shared state.

## Four Ways to Select

Any persistent memory system already has a selection regime, even if the answer is implicit. The authors make it explicit by naming four:

**Ungoverned persistence.** Every plausible summary enters durable storage without evaluation. Low friction, direct path from error to reloadable state. The auto-memory file with three false entries is the textbook failure mode.

**Constitutional or hybrid selection.** Humans write principles or reward models; automated enforcement applies them at scale. RLHF and Constitutional AI illustrate the pattern. Scales well, but can only select for what the principles capture—whatever the principles omit goes ungoverned.

**Automatic metric-based selection.** Benchmarks, tests, or reward signals decide what persists. Powerful when the target is measurable. Weak when you need judgment quality, epistemic honesty, or role diversity—exactly the traits that resist compression into a metric.

**Human-ratified artificial selection.** An operator or governance process directly decides whether a candidate becomes durable shared state. Slower, less scalable, but can select for qualities without clean metrics.

These are not a ranking from worst to best. Each is a design choice over target properties. The question is not whether selection should be automated, but *which traits can be safely selected automatically and which require explicit judgment*.

## Four Layers, Four Governance Burdens

If selection regimes specify *what* becomes durable, the architecture must specify *where* candidate memory lives and how it crosses boundaries.

- **Agent-local memory** — private by default. Preserves the role-specific heuristics, constraints, and failure modes that make one agent useful in a different way from another. Promotable to shared memory only when a record has cross-agent relevance.

- **Shared institutional memory** — durable cross-agent events, principles, decisions, lessons. This is where the strongest governance burden lives, because these records shape behavior across the ecosystem.

- **Archive memory** — long-horizon artifacts and historical records. Retrievable without ratification, but distinguished from active institutional lessons. Prevents governance overload.

- **Project continuity memory** — active task state and handoff context. Temporary and local, preventing scope contamination between transient work and institutional knowledge.

The architecture is not a hierarchy of importance. It is a separation of governance burdens.

## Four Questions That Decide What Persists

The authors distill the governed-memory mechanism into four design questions:

1. **What can vary?** Candidate memories—observed events, distilled principles, protocol changes, failure reports, proposed revisions.

2. **How are candidates evaluated?** They require inspectable evidence and review against current system behavior, not mere plausibility.

3. **Who or what ratifies?** The ratifying authority—human operator, committee, hybrid process—must be explicit, because it determines what becomes durable shared state.

4. **What counts as fixation?** An accepted record enters the shared layer with provenance, version lineage, and *supersede-not-erase* semantics.

That last point matters more than it sounds.

If a false memory is silently overwritten, the system loses the lesson that it failed. If it is superseded with a traceable correction, the correction itself becomes institutional knowledge.

Rejected candidates are not administrative noise—they are evidence that selection occurred.

The authors draw on Ostrom's commons governance: shared memory has boundary, monitoring, conflict-resolution, and correction problems, even though the resource is informational rather than material. Governed collaborative memory instantiates several commons principles—defined boundaries between private and shared memory, monitoring through provenance, graduated ratification requirements, and conflict resolution through explicit judgment.

## What One Ecosystem's Traces Show

The evidence comes from one running ecosystem. The current record: 12 event records (10 ratified, 2 proposed), 8 active principles (all ratified by the ecosystem operator), and a version registry tracking 17 resources and 22 version records.

Five traces carry the argument:

**Ungoverned persistence.** The auto-memory file with three false entries, persisted until audit. No rate is reported—no denominator was tracked.

The point is the failure mode: plausible error becomes reloadable state.

**Governance principle formation.** A procedural lesson was selected and ratified into institutional memory. Documents the *outcome* of selection, not causal superiority.

**Version-registry selection loop.** A candidate registry version was generated, reviewed, found defective, and marked failed. A revised version passed. A subsequent cleanup version recorded the update.

The sequence documents variation, evaluation, rejection, revision, and fixation—making the selection process visible rather than invisible.

**Identity-preserving expansion.** A new agent joined the ecosystem, accessing shared events and principles while maintaining separate role-specific memory. Shared institutional memory grew without merging the new agent's private identity into the common substrate.

**Governance as learning.** This is the most important constraint.

After governance was already in place, an agent fabricated analysis from an unread source. Governance did not prevent the error. Instead, the failure was documented as an event and converted into a ratified *abstention-over-confabulation* principle.

Governance is not a firewall. It is a learning mechanism that converts failures into future selection criteria.

## The Honest Limits

The evidence is observational, from one ecosystem, with no controlled comparison across regimes. The authors are explicit about this. No causal superiority over automatic or ungoverned systems is claimed.

Human ratification creates real bottlenecks and introduces operator bias. An operator who cannot evaluate the relevant domain may ratify the wrong thing. Scaling from one operator to multiple governance actors—especially when ratifiers disagree—remains unresolved.

Legacy memory adds a bootstrapping problem: once a system has operated without governance, older records may already be shaping behavior before their evidence status is known.

The identity-sharing tradeoff is a live tension, not a solved problem. Too little sharing and agents repeat each other's errors. Too much and they converge toward the same generic assistant, losing the cognitive diversity that makes multi-agent systems valuable.

## What This Changes

The paper's contribution is a design agenda, not a benchmark result. It argues that persistent multi-agent systems should evaluate memory not only for recall and performance, but for provenance fidelity, selection traceability, epistemic quality, correction pathways, and role preservation.

A memory system that recalls more facts but obscures *why* those facts became shared state may be less governable than one with lower recall but stronger provenance.

The question shifts from "does memory help?" to "can we inspect, correct, and trust what memory has made durable?"

Every persistent memory system already has a selection regime. Most just hide it inside defaults. This paper makes it visible.

The hallucination wasn't the problem. The heredity was.
